## Committed Items

### Item 33: Job container and services

- Why: Running jobs inside Docker containers and with service containers (databases, caches, etc.) is a common pattern in CI workflows. The external GPT review identified container/services as part of the broader syntax gap. Adding container support enables the SDK to express integration-testing and deployment workflows that require controlled execution environments.
- Prerequisites: Item 28 (workflow-level defaults) should land first because container jobs often combine with workflow-level defaults. ✅ No hard dependency on Item 32 (workflow_call). ✅
- Implementation Plan: Add optional `container` to the step-based job model with `image` (required non-blank string), `credentials` (optional username/password strings), `env` (optional string map), `ports` (optional array of number or string), `volumes` (optional string array), and `options` (optional string). Add optional `services` map to the step-based job model where keys match `^[a-zA-Z_][a-zA-Z0-9_-]*$` and values have the same shape as `container`. Validate required fields (image must be non-blank), validate that numeric ports are positive integers and string ports are non-blank, reject `container`/`services` on reusable-workflow jobs at build time. Render `container` and `services` in canonical job-field positions after `runs-on` and before `outputs`. Update conformance fixtures. Update SPEC.md with canonical orderings before implementation.
- Definition of Done: Step-based jobs support optional `container` with image, credentials, env, ports, volumes, and options. Step-based jobs support optional `services` map with per-service container configuration and identifier-validated keys. Reusable-workflow jobs reject container and services at build time. Validation rejects blank images, invalid ports, and malformed service names. Rendering is deterministic in canonical field order. Conformance fixtures cover container-only, services-only, and combined cases across runtimes. SPEC.md is updated with canonical orderings. The change is code reviewed by a non-implementing persona.
- Acceptance Criteria: Users can define a step-based job with a `container` image and configuration. Users can define service containers for a step-based job. Blank or missing container images fail at build time. Numeric ports must be positive integers; string ports must be non-blank. Service names must match `^[a-zA-Z_][a-zA-Z0-9_-]*$`. Reusable-workflow jobs reject `container`/`services` at build time. Container and services compose correctly with existing job features. Canonical step-based job field order: `if`, `needs`, `continue-on-error`, `permissions`, `timeout-minutes`, `defaults`, `concurrency`, `env`, `strategy`, `runs-on`, `container`, `services`, `outputs`, `steps`. Container field order: `image`, `credentials`, `env`, `ports`, `volumes`, `options`. Credentials field order: `username`, `password`. Cross-runtime conformance fixtures are added.
- Story Points: 3
- Status: done
- Completed At: 2025-07-15
- Notes/Links: [SPEC.md](../SPEC.md). Container and services follow the same explicit-boundary pattern from ADR 0001. Port mapping type is `readonly (number | string)[]`. PR #32 merged into sprint branch.

### Item 35: Validation error diagnostic improvement

- Why: The SDK currently fails explicitly on invalid definitions, but error messages are terse — they state what is wrong without explaining the expected format or suggesting corrections. The external GPT review identified error diagnostic quality as important for both commercial quality and developer experience. Improving error messages reduces the feedback loop for SDK users and makes the validation system more self-documenting.
- Prerequisites: Item 31 (identifier validation) should land first so format-related errors benefit from the improved diagnostic style from the start. ✅ Item 33 (container/services) should land first so the container/services validation surface can adopt the new diagnostic style from the start. No hard dependency on other items.
- Implementation Plan: Audit existing validation error messages across all surfaces (triggers, jobs, steps, strategy, permissions, identifiers, container/services). Enrich messages following the approved convention: `[scope] [field] [constraint]. Expected: [format/values]. [optional fix suggestion]`. Focus enrichment on format validation, unsupported-feature, value-constraint, and type-check messages (~55% of ~106 messages). Enrich blank/empty and boolean check messages only when added guidance is non-trivial. Preserve the existing message prefix structure for programmatic compatibility. Add test coverage for enriched messages. Document the error reporting convention in SPEC.md.
- Definition of Done: Validation error messages include expected formats and valid alternatives where applicable. Error message style follows the approved convention across all validation surfaces. The existing `[scope] [field] [constraint]` prefix structure is preserved for programmatic compatibility. Tests verify enriched messages. SPEC.md documents the error reporting convention. The change is code reviewed by a non-implementing persona.
- Acceptance Criteria: Identifier format errors include the expected pattern `^[a-zA-Z_][a-zA-Z0-9_-]*$`. Type validation errors include the set of valid values (e.g., permission access levels, trigger types, dispatch input types). Permissions errors list valid permission keys when an unknown key is used. Unsupported-feature errors name what is supported instead. Error messages follow the approved convention: `[scope] [field] [constraint]. Expected: [format/values]. [optional fix suggestion]`. Existing message prefix structure is preserved — no breaking change to programmatic error handling. Enrichment targets format, type, value-constraint, and unsupported-feature messages. Blank/empty and boolean checks are enriched only when the added guidance is non-trivial. Error code system is explicitly out of scope. Existing error assertion tests are updated to match enriched messages.
- Story Points: 3
- Status: done
- Completed At: 2025-07-15
- Notes/Links: [SPEC.md](../SPEC.md). This item touches validation messages across all existing surfaces, so the scope is cross-cutting. The implementation must avoid breaking existing programmatic error handling patterns (e.g., `error.issues` array structure). PR #33 merged into sprint branch.
