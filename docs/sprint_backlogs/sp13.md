## Planning Notes

- Capacity decision: Sprint 13 capacity is 16 story points, up from the 15 SP baseline used in earlier sprints to accommodate an in-planning re-estimate of Item 46 from 5 SP to 7 SP. The effective load is 15 SP with a 1 SP intentional buffer.
- Selection decision: Sprint 13 commits `Item 46`, `Item 41`, and `Item 42` for a total of 15 story points. Item 46 covers typed `uses` helpers for both step-level and job-level surfaces. Item 41 introduces the npm build pipeline and `.ts`→`.js` import migration. Item 42 delivers governance and trust documents. Items 43 (README rewrite, 3 SP) and 45 (release automation, 5 SP) are deferred to Sprint 14.
- Ordering decision: Sprint 13 follows the documented backlog priority order Item 46 → Item 41 → Item 42. Item 46 is the highest-priority type-safety item and lands first as a design-first slice with architectural novelty. Item 41 introduces the build pipeline and must land before Item 42's version bump to ensure publish artifacts are in place. Item 42 is mechanical and closes the sprint.
- Buffer decision: The remaining 1 SP is intentionally unallocated. No lower-priority item can be pulled without violating the documented ordering rules.
- Dependency decision: Item 46 has no hard prerequisites (Item 44, its listed prerequisite, delivered in Sprint 12 ✅). Item 41 has no prerequisites. Item 42 has no prerequisites. There are no intra-sprint technical dependencies, but the documented delivery order must be preserved.
- Scope decision: Sprint 13 delivers two major pillars — type-safety deepening (Item 46) and the npm distribution pipeline (Items 41+42). Items 43 and 45 are deferred: Item 43 has a soft prerequisite on Item 41 and is better served once packaging is stable; Item 45 depends on both Items 41 and 42 and forms a natural Sprint 14 opening item.
- Item 46 scope decision: Expanded beyond the original Phase 1 description to cover **both** step-level `uses` (action references) and job-level `uses` (reusable workflow references) in a single sprint slice. Phase 2 (expression helpers) and Phase 3 (typed action wrappers) are explicitly deferred. The two types are `ActionRef` (step-level) and `WorkflowRef` (job-level). This scope expansion drove the re-estimate from 5 SP to 7 SP.
- Item 46 type design decision: `ActionRef` covers three forms — external action reference (`` `${string}/${string}@${string}` ``), local action path (`` `.\/${string}` ``), and Docker image reference (`` `docker://${string}` ``). `WorkflowRef` covers two forms — external reusable workflow (`` `${string}/${string}/.github/workflows/${string}@${string}` ``) and local reusable workflow (`` `./.github/workflows/${string}` ``). Factory functions `actionRef()` and `workflowRef()` provide early TypeScript feedback; `build()` validates all `uses` values at runtime regardless of entry path.
- Item 46 breaking change decision: The `uses` fields on `StepUses` and reusable-workflow job model are changed to pure `ActionRef` and `WorkflowRef` types respectively — no `(string & {})` escape hatch. This is a deliberate breaking change accepted because the project version is still `0.0.0` and `0.1.0` is delivered by Item 42 in the same sprint. Raw string assignment will fail TypeScript type-checking, guiding consumers toward the typed factory functions.
- Item 46 re-estimate decision: Story Points increased from 5 SP to 7 SP during Sprint 13 refinement, reflecting the expanded scope to cover both `ActionRef` and `WorkflowRef` types, two factory functions, breaking-change migration of two model fields, updated conformance fixtures, and SPEC.md updates.
- Item 41 import migration decision: The first sub-task of Item 41 is migrating all 16 `.ts`-extension imports in `packages/` to `.js`-extension imports (TypeScript standard interop style), then removing `allowImportingTsExtensions: true` from `tsconfig.json`. Bun resolves `./foo.js` to `./foo.ts` in source-first workflows, so this is source-runtime-backward-compatible. This migration unblocks `tsc` emit for the npm build step.
- Item 41 build tsconfig decision: Each package (`packages/sdk/`, `packages/cli/`, `packages/core/`) receives its own `tsconfig.build.json` that extends the root tsconfig and adds `declaration: true`, `emitDeclarationOnly: false`, and `outDir: ./dist`. The root `tsconfig.json` retains `noEmit: true` for source-first development.
- Item 41 prepublishOnly decision: `prepublishOnly` script set to `tsc -p tsconfig.build.json` only. No additional smoke-test step at publish time.
- Item 41 CI decision: A build verification step (`tsc -p tsconfig.build.json` in each package) is added to the CI workflow before tests, simulating publish-time verification on every push.
- Item 41 exports decision: `package.json` `exports` field uses the `dist/` output for npm consumers (`import`/`require` conditions); `jsr.json` continues to point to `src/` for JSR/Bun/Deno source-first publishing. The dual-publishing contract is preserved.
- Item 41 CLI shebang decision: The CLI bin entry (`packages/cli/src/bin.ts`) shebang changes from `#!/usr/bin/env bun` to `#!/usr/bin/env node`.
- Item 42 SUPPORT.md decision: Created as a standalone `SUPPORT.md` file in the repository root rather than as a README section.
- Item 42 compatibility policy decision: Documented supported runtimes are Node 22+, Bun 1.x, and Deno 2.x.
- Item 42 CHANGELOG decision: `CHANGELOG.md` uses the Keep-a-changelog format and starts with an `## [Unreleased]` section. `0.1.0` is the first versioned entry, summarizing all Sprint 1–13 deliverables. Release automation (Item 45) in Sprint 14 will wrap this manual CHANGELOG structure.
- Item 42 version bump decision: All `package.json` and `jsr.json` version fields bump from `0.0.0` to `0.1.0`.
- Sprint 12 retrospective carryforward: (a) `displayName()` naming asymmetry documented in LEARN.md, (b) `RunnerLabel` keys use `MacOS` PascalCase (already applied in post-Sprint 12 fixes), (c) SPEC.md clarifies "100% SDK line coverage" with the defensive-branch exemption, (d) capacity baseline: 15 SP baseline with option to raise by ≤ 1 SP during planning when re-estimates justify it (used in this sprint: 16 SP).

### Item 46: Type-safety deepening (`uses` helpers, expression helpers, typed action wrappers)

The review explicitly recommends strengthening type safety beyond runner labels to reduce the gap between "typed workflow builder" and "workflow builder with stringly-typed islands." This item addresses the review's Type-Safety Direction priorities #2–4.
- Prerequisites: Item 44 (typed runner labels) should land first as the simpler type-safety introduction. ✅ Committed in Sprint 12. The broader syntax items (36–40) should also be complete so the type-safe surface applies to the widest possible API. ✅ Committed in Sprint 12.
- Implementation Plan: Deliver `ActionRef` and `WorkflowRef` typed reference types and `actionRef()`/`workflowRef()` factory functions. Update `StepUses.uses` to `ActionRef` and the reusable-workflow job `uses` to `WorkflowRef` (breaking change). Add `build()` runtime validation for all `uses` values regardless of entry path. Add conformance fixtures. Update SPEC.md. Phase 2 (expression helpers) and Phase 3 (typed action wrappers) are explicitly deferred.
- Definition of Done: `ActionRef` and `WorkflowRef` types implemented with template-literal forms covering all documented reference formats. `actionRef()` and `workflowRef()` factory functions implemented with early TypeScript feedback and input validation. `build()` runtime validation rejects malformed references for both model fields. Cross-runtime conformance fixtures updated. SPEC.md updated. Code review completed.
- Acceptance Criteria: SDK consumers construct action references via `actionRef()` and reusable-workflow references via `workflowRef()`. Both factory functions provide compile-time guidance; `build()` rejects malformed values regardless of entry path. `ActionRef` covers: external `owner/repo@ref`, local `./path`, Docker `docker://image`. `WorkflowRef` covers: external `owner/repo/.github/workflows/file@ref`, local `./.github/workflows/file`. The `uses` fields on step and reusable-workflow job models are pure typed refs. Raw string assignment fails TypeScript type-checking. SPEC.md documents the typed-ref API and both type shapes.

### Item 41: Release packaging for npm consumers

- Why: The GPT review identified packaging as the single highest-leverage commercial readiness gap. The current packages ship TypeScript source directly and the CLI requires Bun. Mainstream npm consumers expect built ESM JavaScript with `.d.ts` type declarations and a `node`-executable CLI binary. Without this, adoption by typical Node/npm users is blocked.
- Prerequisites: None (can proceed independently of syntax items).
- Implementation Plan: (1) Migrate all `.ts`-extension imports in `packages/` to `.js`-extension and remove `allowImportingTsExtensions` from `tsconfig.json`. (2) Add per-package `tsconfig.build.json` files that extend the root tsconfig and emit to `dist/`. (3) Update `package.json` exports to point to `dist/` for npm consumers. (4) Change CLI shebang to `#!/usr/bin/env node`. (5) Add `prepublishOnly: tsc -p tsconfig.build.json` to each package. (6) Add CI build verification step. Preserve JSR source-first exports via `jsr.json`.
- Definition of Done: All three packages emit built artifacts into `dist/`. CLI is executable via `node`. `package.json` exports map correctly to `dist/` for npm consumers. JSR publishing continues via `jsr.json` source exports. CI verifies the build step on every push. SPEC.md distribution section updated. Code review completed.
- Acceptance Criteria: `npm pack` produces a tarball with `dist/` containing `.js` and `.d.ts` files. CLI is invocable via `npx @ghawb/cli` without Bun installed. JSR publishing uses `src/` via `jsr.json`. No runtime regression in Bun, Node, or Deno test suites. `allowImportingTsExtensions` removed from root `tsconfig.json`. CI build step passes.
