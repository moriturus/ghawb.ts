## Committed Items

### Item 47: `run` script reference support

- Why: The current `run` step surface only accepts inline command strings, which forces users to manually compose shell-prefixed file execution or inline large script bodies. The requested enhancement adds a first-class way to reference local script files, optionally prefix them with a shell command, and optionally expand the script body into the emitted workflow while keeping the existing explicit-validation architecture.
- Prerequisites: None.
- Implementation Plan: Extend the run-step model and builder API with a script-reference form that supports local file references and shell-prefixed file references in the same slice. Add an optional expand mode that reads the referenced script file and emits its contents as the `run` body. Support both relative and absolute local `path` values. Allow any non-blank `shell` string. Reject double shell specification when both the script-reference config and existing run-step metadata define `shell`. Document Windows quoting as unsupported. Update validation, deterministic rendering, tests, conformance fixtures, and SPEC.md in the same slice.
- Definition of Done: Run steps support local script references, shell-prefixed script references, and optional expand mode. Relative and absolute paths are accepted. Double `shell` specification is rejected at build time. Windows quoting non-support is documented. Tests, renderer coverage, conformance fixtures, and SPEC.md are updated. Code review completed.
- Acceptance Criteria: SDK consumers can author a `run` step from a local script path, optionally provide a shell prefix, and optionally expand the script contents into the emitted workflow. Relative and absolute paths both work for local references. Any non-blank string is accepted for `shell`. If a step defines `shell` in both script-reference config and existing run-step metadata, `build()` fails explicitly. Windows-specific quoting is documented as unsupported for this slice.
- Story Points: 4
- Status: ready

### Item 45: Release automation workflow

- Why: The GPT review's Release Improvement Proposal recommends an automated release workflow covering changesets or equivalent versioning, release PRs, tagged npm/JSR publishing, and GitHub Releases with changelog notes. Without this, every release requires manual version bumps, manual publishing, and manual changelog updates — which is error-prone and scales poorly.
- Prerequisites: Item 41 (release packaging) and Item 42 (governance) should land first so the build pipeline and initial CHANGELOG structure exist before automation wraps them. ✅ Committed in Sprint 13.
- Implementation Plan: Adopt Changesets as the version management system. Add a changeset or release-note requirement to user-facing PRs. Create a release workflow that generates a release PR from `main` with aggregated changelog entries and version bumps. The canonical release state machine is: release PR from `main` → merge release PR → create git tag → publish workflow triggered by the new tag publishes npm and JSR artifacts and generates a GitHub Release with changelog notes. Use a dry-run verification path for sprint closeout proof. Document the release workflow in `RELEASING.md`. Align workflow Node setup with the documented Node 24+ support baseline. Self-host the release workflow using the SDK if feasible.
- Definition of Done: A release workflow exists that automates version bumps, changelog generation, npm publishing, JSR publishing, and GitHub Release creation. The workflow is tested end-to-end via a dry-run path. `RELEASING.md` documents the release flow. SPEC.md distribution section updated. Code review completed.
- Acceptance Criteria: Merging a release PR creates a git tag. The tag event triggers a publish workflow that publishes npm and JSR artifacts and creates a GitHub Release with changelog notes. Version bumps are consistent across all package manifests. The release workflow is documented in `RELEASING.md`. Workflow automation reflects the Node 24+ support baseline.
