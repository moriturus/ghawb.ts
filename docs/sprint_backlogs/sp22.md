## Planning Notes

- Capacity decision: Sprint 22 keeps the standing 20 SP capacity and commits 5 SP, leaving 15 SP residual. The underfill is intentional: the team preserves a package-boundary-focused discovery-first sprint rather than mixing in unrelated hardening work just to consume capacity.
- Selection decision: Sprint 22 commits Items 75a and 75b for a total of 5 SP. Item 77 remains in the product backlog as the next deferred item.
- Ordering decision: Sprint 22 preserves strict backlog execution order: `Item 75a` -> `Item 75b`.
- Dependency decision: `Item 75b` does not begin unless `Item 75a` is review-complete and explicitly recommends migration away from the current `@ghawb/sdk` placement. If `Item 75a` recommends retaining the current boundary, Sprint 22 stops after `Item 75a` and does not pull `Item 77` into the sprint.
- Item 75a discovery decision: The discovery slice compared three options for the future of high-level job helpers: keep `nodeCi()` in `@ghawb/sdk`, move it to a new opt-in package, or move it into an existing ergonomics-oriented package. The initial recommendation was to retain in SDK, but the Product Owner overrode that decision based on prior team consensus that `nodeCi()` is too specific for the core SDK. Corrected outcome: migrate `nodeCi()` to a new opt-in package `@ghawb/job-helpers`; original decision recorded in [ADR 0005](../adrs/0005-retain-node-ci-helper-in-sdk.md) (now superseded), corrected decision recorded in [ADR 0006](../adrs/0006-migrate-node-ci-helper-to-opt-in-package.md).
- Item 75b scope decision: The implementation slice must stay minimal: move only the approved helper surface to `@ghawb/job-helpers`, provide explicit migration guidance, update affected docs, and avoid widening into a broader preset framework.
- Collaboration decision: Sprint execution remains single-item-at-a-time in committed order. Sub-agent or multi-agent collaboration is preferred only inside the active item when it improves separation of concerns. Primary implementation responsibility: Mio Kanda (`Items 75a` and `75b`). Verification and review focus: Haru Nishimura. Product-surface acceptance and scope judgment: Aoi Sakamoto. Tooling and documentation friction support: Yui Morita. Planned review ownership: Aoi Sakamoto reviews `Item 75a` for product-direction fit; Haru Nishimura reviews `Item 75b` for migration correctness, verification coverage, and contract safety.
- Verification decision: Sprint 22 continues to use targeted `bun test` commands for item proof, `bun run check` as the primary integrated quality gate, Node-run Vitest as the secondary runtime check, and `bun run verify:workflows` when self-hosted workflow sources change as part of the approved migration slice. Deno remains compatibility-oriented coverage rather than the primary sprint test runner unless the migration decision directly changes that boundary.
- Sprint branch: `sprint-22` from `main`; pushing the branch to `origin` remains required before execution starts.

## Committed Items

### Item 75a: Discover package boundary and migration plan for high-level job helpers

- Why: `nodeCi()` is currently an accepted narrow helper inside `@ghawb/sdk`, but the backlog still contains pressure to move high-level job helpers behind an opt-in package boundary. That change now touches shipped docs, self-hosted workflow guidance, and the repository's recent hardening work, so a fresh discovery slice is needed before implementation reopens the package-boundary decision.
- Prerequisites: None. Discovery should treat the current `nodeCi()` placement as shipped behavior and should explicitly incorporate the Sprint 19 discovery outcome plus Sprint 21 planning constraints.
- Implementation Plan: Compare at least three options for the future of high-level job helpers: keep `nodeCi()` in `@ghawb/sdk`, move it to a new opt-in package, or move it into an existing ergonomics-oriented package if that boundary remains coherent. Evaluate migration cost, self-hosted workflow impact, public API stability, documentation churn, and whether the move improves package coherence enough to justify the change. Produce a decision-ready recommendation plus the smallest safe implementation slice if migration is approved.
- Definition of Done: A documented recommendation exists for the preferred package boundary and migration direction, rejected alternatives are summarized briefly, and any follow-up implementation slice is backlog-ready with no major open design questions.
- Acceptance Criteria: Discovery explicitly compares the current `@ghawb/sdk` placement against at least two opt-in-boundary alternatives, states whether migration should proceed, explains the impact on self-hosted workflow sources and public docs, and leaves the follow-up implementation item with a concrete migration path or an explicit decision to retain the current boundary.
