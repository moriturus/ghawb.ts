Developer ownership should be split by concern, with Mio Kanda leading the CLI/render-contract slices (`Items 81`, `82`) and Yui Morita leading the repository-convention and config-manifest support slices (`Items 79`, `80`, `83`) under Product Owner scope control. Planned review ownership should stay with a different persona than the implementation owner for each change.
- Verification decision: Sprint 24 should continue to use `bun run check` as the primary integrated quality gate, Node-run Vitest as the compatibility confirmation path, and workflow-guardrail verification if workflow source files change. CLI tests must cover detection, diagnostics, and config-manifest behavior from the user-facing command perspective.

Regenerate affected workflow YAML and update any specification text that still embeds older defaults.

### Item 81: Fold `render-batch` into `render`

- Why: The current CLI splits workflow rendering across `render` and `render-batch`, but the two commands differ only in whether they accept one target or multiple explicit `--input` / `--output` pairs. That extra command surface increases learning cost without providing a distinct product capability.
- Prerequisites: None.
- Implementation Plan: Extend `ghawb render` so it accepts either one target or multiple explicit `--input` / `--output` pairs, preserve the existing single-target output-path inference only when exactly one target is rendered, and remove `render-batch` as a compatibility path so the canonical rendering contract is a single command.
- Definition of Done: The primary workflow-rendering contract is a single `render` command that supports both single-target and multi-target authoring flows, documentation is updated, and the change completes with code review.
- Acceptance Criteria: `ghawb render -i a.ts -o a.yml -i b.ts -o b.yml` succeeds as the supported multi-target path; single-target inference for `workflows/<name>.ts` still works only when a single render target is present; `render-batch` is removed rather than preserved as a compatibility path; tests cover direct multi-target rendering; docs and SPEC describe `render` as the canonical command.
- Story Points: 2
- Status: ready
- Completed At: N/A
- Notes/Links: Intake created from Product Owner follow-up after Sprint 23 discovery. Rationale: batch-specific command surface is redundant when rendering semantics remain otherwise identical. See [Sprint 23 Review](../sprint_reviews/sp23.md) and [Specification](../SPEC.md).

### Item 82: Unify `render-action` under `render` with safe export auto-detection

- Why: The command surface is easier to use when workflow and composite-action rendering are converged behind one `render` entrypoint, but the user still needs clear, dependable behavior when the CLI decides which artifact type a module represents.
- Prerequisites: Item 81. The workflow rendering contract should be unified first so action unification builds on the final `render` shape rather than on a superseded command split.
- Implementation Plan: Design and implement a single-entry rendering contract for workflows and composite actions using safe export-type auto-detection as the supported path. Preserve existing validation clarity, keep workflow-only output inference scoped to supported workflow sources, and provide explicit diagnostics when a module cannot be classified safely.
- Definition of Done: `render` can handle both workflow and composite-action rendering through one reviewed contract, the auto-detection strategy is documented and tested, backward compatibility is handled explicitly for the unified command surface, and the change completes with code review.
- Acceptance Criteria: The unified command makes it clear whether the input is treated as a workflow or composite action without requiring a separate kind flag; invalid or ambiguous inputs fail with explicit diagnostics; workflow output inference remains limited to the documented `workflows/<name>.ts` path; composite-action output handling remains explicit in the rendered result; docs and SPEC reflect the new canonical CLI.
- Story Points: 6
- Status: ready
- Completed At: N/A
- Notes/Links: Intake created from Product Owner follow-up after Sprint 23 discovery. The supported path is now safe export auto-detection because that is the most direct user-facing contract for a single `render` entrypoint. See [Sprint 23 Review](../sprint_reviews/sp23.md) and [Specification](../SPEC.md).

### Item 83: Design CLI config-manifest support for workflow and action rendering

- Why: Users want to inject render configuration through JSON, YAML, or TOML files, but the CLI does not yet define a stable config schema, precedence order, or command-scope boundary for that capability. The config contract should support the same user-facing render surface that `render` now exposes.
- Prerequisites: Item 81. Item 82 informs the final scope, and the first config slice must support both workflow and composite-action rendering so the config entry point does not split the user story back apart.
- Implementation Plan: Define a CLI-owned config schema and precedence model for render commands, covering allowed target declarations, format support (JSON / YAML / TOML), flag-overrides-config behavior, and explicit out-of-scope cases. Keep the responsibility at the CLI edge rather than expanding the SDK or core renderer into a general config-loading layer.
- Definition of Done: A reviewed CLI config contract exists for render flows, the scope boundary is documented in SPEC and CLI-facing docs, and the chosen contract is ready to implement without reopening the command-surface decisions from Items 81 and 82.
- Acceptance Criteria: The config design states how multiple targets are declared, how JSON/YAML/TOML inputs are interpreted, what wins between CLI flags and config values, how both workflow and composite-action rendering are represented, and which invalid shapes fail explicitly; the design keeps config parsing at the CLI boundary rather than in the SDK.
- Story Points: 4
