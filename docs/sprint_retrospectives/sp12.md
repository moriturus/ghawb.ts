# Sprint Retrospective: Sprint 12

## Summary

Sprint 12 retrospective covers the delivered slice `Item 36`, `Item 37`, `Item 38`, `Item 39`, `Item 40`, and `Item 44` (18 story points out of 20 capacity). This was the first sprint following Sprint 11's "product backlog empty" close; the backlog was repopulated via a second GPT review intake before Sprint 12 started.

- All 6 items were delivered in the planned execution order without resequencing: display names, environment, pull_request_target, workflow_run, simple events, typed runner labels.
- Sprint 12 ran at 18 SP — a 50% increase over the 12 SP delivered in Sprints 10 and 11. Delivery succeeded because all 6 items followed the same established model/builder/validation/rendering/conformance pattern with no architectural novelty.
- The Sprint 11 retrospective carryforwards (format-before-verify, temp file cleanup, file ownership coordination) were incorporated into the Sprint 12 planning notes and delegation prompts.
- Sprint closeout completed end-to-end: the sprint branch was merged into `main` through sprint PR [#43](https://github.com/moriturus/ghawb.ts/pull/43).
- Evidence provenance: retrospective was performed against a clean worktree on `main` at commit `3ee0029`.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent AST design, builder contracts, renderer ordering, and API compatibility.
- Observed that the six items confirm the pattern is fully stable. Every item followed the same workflow: update model, add builder method, add validation, add renderer entry, add conformance fixtures, update SPEC.md. No item required any pattern deviation.
- Observed one API naming inconsistency surfaced during the sprint review: the workflow-level `run-name` field is set via `.runName()` on the workflow builder, but the job-level `name` field is set via `.displayName()` on the job builder — not `.name()`. The `displayName()` naming exists to avoid a collision with the built-in `Function.name` property (since builders are class instances). This is defensible but may cause discovery friction for SDK consumers who try `.name()` first. It should be documented in SPEC.md and LEARN.md.
- Observed that Item 44's `RunnerLabel` uses the casing `Macos` (not `MacOS`). This diverges from Apple's conventional styling and may cause minor friction for developers expecting `MacOS14` instead of `Macos14`. The string values are correct (`macos-14`, etc.); only the key names are affected. Given that backward compatibility governs these constants once published, this should be evaluated before any first release.
- Wants the `RunnerLabel` casing question resolved explicitly in a backlog item before the project releases, so the constants are not locked in with inconsistent key names.

## Grouped Improvements

### Team Improvement

1. **Job builder `displayName()` naming**: The job name builder method uses `displayName()` to avoid a conflict with `Function.name`, but SDK consumers may expect `.name()` by analogy with the workflow-level `.runName()` naming pattern. Add a LEARN.md entry documenting this naming decision and its rationale so the pattern is explicit for future SDK surface additions.

2. **`RunnerLabel` key casing before publish**: `RunnerLabel` keys use `Macos` (e.g., `Macos14`) rather than `MacOS` (e.g., `MacOS14`). This should be evaluated and resolved before the first npm/JSR release, since published constant names become a breaking-change boundary. Raise this as a scope item during Item 41 (release packaging) planning.

3. **Branch coverage documentation**: The persistent gap between 100% line coverage and ~97–98% branch coverage (unreachable defensive branches in type-narrowed union code) is consistent across multiple sprints. Document this known exception formally in SPEC.md under the quality constraints section to avoid recurring re-explanation in sprint reviews.

4. **Capacity baseline discipline**: Sprint 12 delivered 18 SP on a 20 SP capacity, which was 50% above historical velocity. This should not be treated as a new capacity baseline. Sprint 13 planning should explicitly re-evaluate capacity given Item 46 is a design-first, architecturally novel item — not a pattern-uniform syntax expansion.

### Product Improvement

1. No new product backlog item is required from Sprint 12. The delivered increment matched the intended scope and no confirmed correctness defects were found during review.
2. The `RunnerLabel` casing concern (team improvement #2) should be evaluated during Item 41 planning. If a correction is needed, it can be included in Item 41 or added as a small scope item in the same sprint before release.
