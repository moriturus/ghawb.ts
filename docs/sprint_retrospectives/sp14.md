# Sprint Retrospective: Sprint 14

## Summary

Sprint 14 delivered all three committed items totaling 12 SP: Item 47 (run script references, 4 SP), Item 43 (README rewrite, 3 SP), and Item 45 (release automation, 5 SP). The sprint closed with PR #51 into `main`. A post-merge CI fix was required for a `tsc` strict compilation error.

- Item 47 added `runScript()` with local file references, shell prefixes, and expand mode.
- Item 43 rewrote the README for external consumers with five verified examples.
- Item 45 adopted Changesets and added self-hosted release and publish workflows.

Two new backlog items were surfaced: Item 48 (reusable workflow object injection) and Item 49 (npm install `workspace:*` fix).

## Role Reflections

### Product Owner — Aoi Sakamoto

- Role-based viewpoint: Aoi can credibly represent scope control, backlog intake, and acceptance decisions.
- Observed that the sprint-level scope was clean — all three items shipped without scope churn or deferral. The mid-sprint replan to insert Item 47 before Item 43 worked well; the new SDK surface informed the README examples.
- Observed that two new backlog items emerged during sprint execution (Items 48 and 49). Both are genuine product needs: Item 49 blocks npm consumers, and Item 48 improves SDK ergonomics.
- Wants Item 49 prioritized ahead of Item 48 in the backlog since `npm install` failure is a blocking issue for npm users. Wants the next sprint to balance bug fixes with new feature work.

### Scrum Master — Ren Takahashi

- Role-based viewpoint: Ren can credibly represent flow, coordination, and process discipline.
- Observed that the branch-per-item, PR-per-item path remained stable across all three items and the sprint closeout.
- Observed that the CI failure was caught only after the sprint-level PR was created. The root cause — Bun accepting type mismatches that `tsc` rejects — is a systemic risk. The fix was straightforward but cost an extra iteration.
- Wants a local `tsc` check added to the development workflow (pre-push hook or explicit verification step) so type mismatches are caught before CI. This is recorded in `docs/LEARN.md` but should become an enforced practice.
- Wants the scrum board updated if the `tsc` pre-push check becomes a tracked process improvement.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent API design, model extension, and runtime compatibility.
- Observed that the `runScript()` API landed cleanly with the reference/expand duality. The `ScriptReference` model type preserves explicit-boundary principles from ADR 0001.
- Observed that removing dead validation code (blank path/shell checks that became unreachable after early-throw) was the right call for coverage discipline. Defense-in-depth is valuable, but unreachable code undermines the 100% coverage goal.
- Wants future validation design to choose a single canonical enforcement point for each invariant rather than duplicating checks across early-throw and build-time paths.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent verification rigor, test design, and coverage discipline.
- Observed that the README example tests (`tests/node/readme-examples.test.ts`) are a strong pattern — verifying documentation examples compile and render valid YAML catches documentation rot at CI time.
- Observed that the CI failure exposed a gap: the local `bun test` + `bun run coverage` loop does not catch type-level errors that only `tsc` strict mode rejects. This is the same class of gap documented in LEARN.md.
- Wants the standard verification command (`bun run check`) to include `tsc` compilation so the local loop matches CI exactly. Currently `bun run check` runs lint, test, coverage, format, and Deno tests, but `tsc` is only in CI.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent repository ergonomics, tooling, and packaging.
- Observed that the self-hosted release and publish workflows demonstrate the SDK eating its own dog food effectively. The `workflows/` directory convention with auto-discovery works well.
- Observed that `npm install` fails due to `workspace:*` — this was introduced in earlier sprints but only caught now because the team primarily uses Bun. The npm compatibility claim in the README is currently broken for local development.
- Wants the `workspace:*` fix (Item 49) to include a CI verification step that runs `npm install` to prevent regression.

## Team Improvements

1. **Add `tsc` to local verification**: Include `tsc -p packages/*/tsconfig.build.json` in `bun run check` so the local development loop catches the same type errors CI does. The Bun-vs-tsc gap has now caused one CI failure and is documented in LEARN.md as a recurring risk.
2. **Single canonical enforcement point per invariant**: When adding validation to builder methods, choose either early-throw or build-time validation for each invariant, not both. Unreachable validation code undermines the 100% coverage goal.
3. **Cross-package-manager CI verification**: Add an `npm install` step to CI (or a periodic check) to verify that npm compatibility claims in the README remain true.

## Product Improvements

1. **Item 49 — npm install fix** (2 SP): Fix `workspace:*` protocol incompatibility so `npm install` and `npm ci` work. Prioritize ahead of new feature work.
2. **Item 48 — reusable workflow object injection** (3 SP): Allow `usesWorkflow()` to accept `WorkflowBuilder` or `WorkflowDefinition` directly with `workflow_call` validation at the caller's `build()` time.
