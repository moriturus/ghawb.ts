# Sprint Retrospective: Sprint 15

## Summary

Sprint 15 delivered all three committed items totaling 6 SP: Item 49 (fix npm workspace: protocol, 2 SP), Item 50 (close tsc build-config gap, 1 SP), and Item 48 (reusable workflow object injection, 3 SP). The sprint closed with PR #55 into `main`. No CI failures, no mid-sprint replans, no scope churn.

Sprint 15 was the smallest sprint by committed load (6 SP against 20 SP capacity) across the project's 15-sprint history. The 14 SP buffer reflected genuine backlog exhaustion — all prior items were delivered and no forced intake was appropriate. The mix of work (bug fix, process improvement, SDK feature) was set by explicit PO priority from the Sprint 14 review.

With this sprint complete, 52 product backlog items have been delivered across 15 sprints. The product backlog is empty.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the Item 48 API design and the `ReusableWorkflowSource` type architecture.
- Observed that the `ReusableWorkflowSource` union type follows the project's established pattern of discriminated unions with runtime detection: `typeof === "string"` → `instanceof WorkflowBuilder` → object fallback. This is consistent with how the renderer dispatches on trigger types and job variants.
- Observed that deferring `workflow_call` validation to `build()` time rather than `usesWorkflow()` call time was the correct design decision. It enables definition-order independence, which is a real ergonomic win for multi-workflow repositories where the reusable workflow may be defined after the caller.
- Observed a minor code quality note: the `usesWorkflow()` implementation has a defensive `else` branch for null/undefined that TypeScript's type system prevents from being reached. This is harmless defense-in-depth but could be removed for clarity. Not worth a follow-up item.
- Wants future API extensions to continue the pattern of build-time validation over call-time validation when definition-order independence matters.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent test design, coverage discipline, and verification rigor across all three items.
- Observed that the `build:check` addition (Item 50) closes the most significant verification gap the project has had. The local loop now matches CI exactly for type checking. No new Bun-vs-tsc incidents should occur.
- Observed that Item 48's test suite covers a complete decision matrix: two source types × happy/error paths × with/secrets forwarding × definition-order independence × backward compatibility = 7 tests. The conformance fixture for rendered output adds cross-runtime verification. This is thorough.
- Observed that 100% line, function, and statement coverage was maintained (100% line, 98.27% branch). The branch gap is pre-existing and relates to defensive branches in the codebase, not Sprint 15 changes.
- Wants to note that the test count trajectory remains healthy: 362 Vitest + 76 Deno = 438 total tests, up from 429 (354+75) in Sprint 14.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent the npm workspace fix (Item 49) and CI tooling changes.
- Observed that replacing `workspace:*` with `^0.1.0` semver ranges was simpler and more robust than the alternatives (file: paths, npm workspaces restructuring). The key insight was that all package managers resolve workspace packages through the root `workspaces` field when the version range is satisfied, so `^0.1.0` works universally.
- Observed a gotcha worth remembering: npm's arborist crashes when encountering bun's `.bun/` directory layout inside `node_modules`. A full `rm -rf node_modules packages/*/node_modules` was needed before `npm install` would work cleanly. This is not documented anywhere obvious.
- Observed that committing `package-lock.json` was necessary for `npm ci` but adds a maintenance burden — it must stay in sync when dependencies change. The current workflow (primarily bun install, npm for verification) means the lockfile could drift silently.
- Wants a LEARN.md entry for the bun/npm node_modules incompatibility gotcha.
