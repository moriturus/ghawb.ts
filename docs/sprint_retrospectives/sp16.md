# Sprint Retrospective: Sprint 16

## Summary

Sprint 16 delivered all five committed items totaling 19 SP: Item 51 (expression helper MVP, 5 SP), Item 52 (advanced runs-on support, 3 SP), Item 53 (actionlint bridge, 3 SP), Item 54 (API reference and cookbook docs, 3 SP), and Item 55 (opt-in reusable-workflow import package, 5 SP). The sprint closed with PR #64 from `sprint-16` into `main`. No CI failures, no mid-sprint replans, no scope churn.

Sprint 16 was the largest sprint since Sprint 14 (19 SP vs 6 SP in Sprint 15) and the broadest in scope mix: SDK feature extensions (Items 51, 52), CLI tooling (Item 53), user-facing documentation (Item 54), and a new opt-in package (Item 55). The sprint exhausted all items remaining from the product discovery that preceded it. With Sprint 16 complete, 57 product backlog items have been delivered across 16 sprints. One new item (Item 56) was added to the backlog from PO feedback during the sprint review.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the expression helper API design (Item 51), runs-on object form architecture (Item 52), and reusable-workflow import package architecture (Item 55).
- Observed that the expression helper design (Item 51) followed the project's established pattern of composable string builders: context helpers produce naked reference strings (`github.ref`), and `expr()` wraps them in `${{ }}`. This is consistent with how the renderer separates structural construction from string formatting.
- Observed that Item 52 exposed a Deno-specific type narrowing strictness issue: inline `typeof`/`Array.isArray` checks do not narrow union types the same way in Deno's type-checker as in Bun/Node TypeScript. The workaround (explicit `isRunsOnObject()` type guard with annotated return type) is architecturally clean and already documented in LEARN.md. This pattern should be the default approach for future union narrowing.
- Observed that the reusable-workflow import package (Item 55) cleanly preserved the core/opt-in boundary per ADR 0001. The `@ghawb/sdk` package has zero YAML parsing code. The `@ghawb/reusable-workflow-import` package depends on the `yaml` library internally but does not re-export it. The boundary is explicit and testable.
- Observed that the `ImportDependencies` DI pattern in `@ghawb/reusable-workflow-import` follows the same approach as `CliRunDependencies` in the CLI package. This consistency makes the testing patterns predictable across packages.
- Wants future packages to continue the DI-for-testability pattern rather than relying on test-time mocking of module internals.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent test design, coverage discipline, and verification rigor across all five items.
- Observed a significant test growth: 362 → 444 Vitest tests (+82) and 76 → 78 Deno conformance tests (+2). The Vitest growth reflects the breadth of Sprint 16 (52 expression tests, 10 runs-on tests, 5 lint tests, 13 reusable-workflow-import tests, plus fixture updates).
- Observed that 100% line, function, and statement coverage was maintained across 444 tests. Branch coverage remains at 98.35% — the gap is pre-existing defensive branches, not Sprint 16 changes.
- Observed a latent test-helper bug in Item 55: the `createTempYaml()` helper always wrote to a fixed filename (`shared-build.yml`), causing three tests that expected different basenames to fail. The fix was straightforward (use per-test filenames), but the bug illustrates that test helpers with implicit assumptions can create confusing failures when tests are written independently.
- Observed that conformance fixtures were updated for both expression helpers (Item 51) and runs-on object form (Item 52), maintaining the cross-runtime verification contract. Deno conformance remains a lightweight smoke layer (78 tests vs 444 Vitest), which is appropriate for the current project scale.
- Wants to recommend that future test helpers accept explicit parameters for values they currently hardcode (filenames, paths, identifiers) to avoid the class of latent bug seen in Item 55.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent CLI tooling (Item 53), documentation (Item 54), and root configuration management.
- Observed that the CLI lint command (Item 53) followed the established DI pattern from the existing render commands. The `findExecutable` and `runCommand` dependencies were added to `CliRunDependencies` cleanly. The non-zero-exit behavior for missing `actionlint` was straightforward to implement and test.
- Observed that the API reference and cookbook (Item 54) fill a real gap identified in the Sprint 15 retrospective. The cookbook's 11 recipes cover the most common workflow patterns and provide a clear bridge from the README to deeper SDK usage. The explicit `displayName()` gotcha documentation addresses the LEARN.md recommendation from Sprint 12.
- Observed that adding `@ghawb/reusable-workflow-import` as the fourth package required updating five root configuration files (vitest.config.ts, tsconfig.json, deno.json, jsr.json, package.json). Each update is small, but missing any one of them causes silent test or resolution failures. This ceremony should be documented as a checklist in LEARN.md if more packages are anticipated.
- Wants to note that the `import` naming conflict in `cli.test.ts` (local `runCli` helper vs imported `runCli` from the module) was a minor friction point. Aliasing the import to `runCliDirect` resolved it, but the pattern could recur if test files grow to import more of their own module's exports.

## Team Improvements

1. **Document new-package checklist**: Adding a package to the monorepo requires coordinated updates across 5 root config files (vitest.config.ts, tsconfig.json, deno.json, jsr.json, package.json). This should be recorded as a checklist in LEARN.md to prevent silent failures when future packages are added.
2. **Test helper parameterization**: The `createTempYaml()` latent bug (hardcoded filename) demonstrates that test helpers should accept explicit parameters for values they embed. This is a minor quality discipline improvement, not a process change.
3. **Sub-agent delegation review**: Sprint 16 used minimal sub-agent delegation despite the working agreements encouraging it. For future sprints with 4+ items, consider whether delegating independent concerns (e.g., documentation vs implementation) to sub-agents would improve throughput within the sequential-item constraint.

## Product Improvements

1. **Release evaluation**: The project is in a release-ready state after 16 sprints. The PO recommends evaluating a 0.1.0 release before adding more features. This is a product priority decision, not a backlog item.
2. **Item 56 (string shorthand)**: Already captured in the product backlog. A small ergonomic improvement (2 SP) that could ship alongside or after release preparation.
3. **Expression helper expansion**: The MVP covers context references and status-check functions. Future expansion could add operator helpers, ternary expressions, or format functions. This would require a new product discovery intake.
4. **Deno type guard pattern**: The `isRunsOnObject()` workaround from Item 52 should become the standard approach for union narrowing across the codebase. This is a code quality pattern, not a backlog item — enforce through review conventions.
