# Sprint Retrospective: Sprint 18

## Summary

Sprint 18 delivered all four committed items totaling 13 SP: Item 61 (typed action wrappers, 5 SP), Item 62 (CLI `render`/`render-batch` `--lint` integration, 2 SP), Item 63 (reusable workflow caller-side outputs, 3 SP), and Item 64 (expression helper expansion, 3 SP). The sprint closeout PR #74 from `sprint-18` into `main` is open, and no committed scope carried over.

Sprint 18 was a feature-maturation sprint. It resumed feature delivery after Sprint 17's release-readiness work and completed all selected items in backlog order without mid-sprint replanning, CI failures, or acceptance disputes.

Evidence provenance: evidence from clean committed snapshot at `sprint-18` branch HEAD (`6345502`).

## Role Reflections

### Product Owner — Aoi Sakamoto

- Role-based viewpoint: Aoi represents scope control, acceptance criteria enforcement, and product-priority judgment across all four delivered items.
- Observed that Sprint 18's scope cuts were effective. Item 61 stayed manual-first, Item 63 avoided invalid caller-side YAML expansion, and Item 64 stayed within comparison/logical operators rather than growing into a full expression language project.
- Observed that the product strategy choice to defer `0.1.0` and continue feature maturation did not create churn inside the sprint. The team executed the selected slice cleanly, and the delivered surface area is coherent.
- Observed that the active backlog is now effectively exhausted from an implementation-readiness perspective. Item 65 is the only remaining active item, but it is still discovery-bound rather than sprint-ready.
- Wants Sprint 19 preparation to start with discovery, not implementation. Item 65 should explicitly evaluate whether the right answer is an SDK preset API, Cookbook-first guidance, or a narrower hybrid.

### Scrum Master — Ren Takahashi

- Role-based viewpoint: Ren represents flow quality, sprint discipline, coordination, and role-boundary enforcement.
- Observed that Sprint 18 followed the repository execution contract cleanly: one backlog item at a time, feature branches from `sprint-18`, item PRs into `sprint-18`, item-level review and PO acceptance before merge, then sprint-level PR #74 into `main`.
- Observed that the sprint had no meaningful impediments. Each item was small enough to move from implementation to review quickly, and repeated `bun run check` verification kept risk low.
- Observed that the repository asks agents to prioritize sub-agents or multi-agent collaboration during sprint execution, but Sprint 18 did not materially benefit from that pattern. The committed work was sequential, API-scoped, and small enough that forced delegation would likely have added coordination overhead rather than reducing cycle time.
- Wants the team guidance to state this more explicitly: sub-agent or multi-agent collaboration should be preferred when it helps, not performed ritualistically when the current top item is too small or too coupled to justify it.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the design and implementation choices for Items 61, 63, and 64.
- Observed that the additive API pattern held up again. Typed wrappers, reusable-workflow output metadata, and expression operator helpers all landed by extending the public surface without forcing internal rewrites or backward-incompatible changes.
- Observed that Item 63's internal-metadata approach was the correct architectural cut. Caller-side reusable workflow outputs belong in the SDK model for validation and authoring help, but they do not belong in rendered reusable-workflow caller YAML.
- Observed that Item 64 needed `literal(...)` alongside operator helpers to be genuinely useful. Comparison helpers without a quoting strategy would still force raw string assembly at call sites.
- Wants Item 65 discovery to preserve the same architectural bias: prefer explicit additive helpers over magical job templates that hide the existing builder model.

- Role-based viewpoint: Yui can credibly represent the CLI/tooling slice from Item 62 and the documentation-sync experience across the sprint.

- Observed that README and `docs/` stayed aligned during Sprint 18 because documentation updates were made inside each item rather than deferred to a final cleanup pass. That reduced the chance of another feature-status mismatch.

## Team Improvements

1. **Clarify sub-agent usage threshold**: TEAM and/or Scrum Master guidance should say that sub-agent or multi-agent collaboration is preferred when it materially improves throughput or separation of concerns, but should not be forced on small, tightly coupled, sequential items. Sprint 18 validated the sprint workflow without meaningful delegation, and that should be considered acceptable rather than a process miss.

2. **Keep sprint-review demo evidence item-local**: The sprint review was efficient because each item already had a narrow verification command that demonstrated the delivered increment. This should become an explicit closeout expectation: every delivered item should leave behind one minimal demo command or proof artifact that can be reused in the sprint review.

## Product Improvements

1. **Item 65 discovery must compare API vs. documentation-first alternatives**: The current backlog framing for job presets assumes an SDK surface may be needed, but Sprint 18 reinforced the value of additive, explicit APIs. Discovery for Item 65 should explicitly compare at least three options: Cookbook-only guidance, a narrow helper API, and a broader preset/recipe layer.

2. **Keep feature maturation additive**: Sprint 18's best outcome was that all four items extended the product surface without destabilizing the architecture. Future product work should continue preferring explicit opt-in helpers and metadata over hidden rendering behavior or broad abstraction layers.
