# Sprint Retrospective: Sprint 19

## Summary

Sprint 19 delivered both committed items totaling 5 SP: Item 65 (job preset / recipe discovery, 2 SP) and Item 66 (job preset / recipe initial implementation slice, 3 SP). The sprint closed with PR #77 from `sprint-19` into `main`, and no committed scope carried over.

Sprint 19 was a discovery-first validation sprint. It split one ambiguous theme into a documented design decision plus one narrow implementation slice, then shipped that slice as `JobBuilder.nodeCi()` in `@ghawb/sdk` without broadening into a larger preset framework.

Evidence provenance: evidence from the clean `main` snapshot after Sprint 19 closeout, including merged PRs #75, #76, and #77 plus the recorded Sprint 19 review note.

## Role Reflections

### Product Owner — Aoi Sakamoto

- Role-based viewpoint: Aoi represents scope control, acceptance-criteria judgment, and product-priority decisions across the discovery and delivery slices.
- Observed that splitting the original job-preset idea into discovery first and implementation second was the right call. Item 65 closed the major design questions before the team spent effort on API code.
- Observed that the selected narrow-helper outcome matched the project's additive-API bias and avoided a premature commitment to a generic recipe framework.
- Observed that Sprint 19's low committed load was intentional and justified. The team used underfill to preserve delivery quality rather than pulling in unrelated backlog items.
- Wants future recipe-like work to require the same level of evidence: repeated real patterns first, then the smallest additive helper that proves value.

### Scrum Master — Ren Takahashi

- Role-based viewpoint: Ren represents delivery flow, branch/PR discipline, impediment removal, and process compliance.
- Observed that the item-by-item sprint flow worked as intended once execution was underway: Item 65 merged into `sprint-19`, then Item 66 started from the updated sprint branch, then the sprint closeout PR went to `main`.
- Observed two small but real process gaps:
  - the sprint branch existed locally before execution, but had not been pushed to `origin` before the first item PR was opened, which caused avoidable base-branch friction
  - Item 65 was briefly marked `done` in the sprint backlog before durable PR/review evidence existed, and the gap had to be corrected before proceeding
- Observed that sub-agent use was productive for role-specific evidence gathering and review support in this sprint because the active work was small but the design judgment and proof collection still benefited from separation of concerns.
- Wants the sprint-start and item-closeout protocols to make the remote sprint-branch check and review-evidence gate more explicit.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the design and implementation choices for ADR 0002 and `JobBuilder.nodeCi()`.
- Observed that the best part of Sprint 19 was architectural restraint. The team proved a useful repeated pattern without adding a new package, a workflow-template layer, or hidden behavior.
- Observed that `nodeCi()` fits the existing builder model because it appends explicit steps rather than replacing the builder with a separate declarative DSL.
- Observed that discovery and implementation stayed aligned because ADR 0002 fixed the package boundary and public-shape expectations before code started.
- Wants future helper work to preserve this boundary: explicit builder additions, no silent defaults outside the helper's local scope, and no assumption that one helper implies a full preset framework.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent verification quality and review rigor for the implementation slice.
- Observed that the sprint stayed reviewable because Item 66 added tests before widening docs/examples, and `bun run check` remained the stable integrated quality gate.
- Observed that the review process caught a real execution issue on Item 65: backlog status had moved to `done` before durable review/acceptance evidence was recorded. The process recovered cleanly, but the rule should be harder to violate.
- Observed that the implementation slice is acceptance-safe because the helper has both success-path tests and failure-path tests for blank inputs.
- Wants the closeout checklist to force a final "evidence exists before status flip" check for each item, especially for documentation-heavy items where it is easy to confuse decision-complete with review-complete.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent branch/PR ergonomics, documentation synchronization, and workflow friction.
- Observed that the docs remained aligned because Item 66 updated README, Cookbook, API reference, and SPEC in the same implementation slice rather than deferring documentation cleanup.
- Observed that the only real tooling friction in the sprint was operational: item PR creation assumed the sprint branch already existed on `origin`, but that assumption was not verified up front.
- Observed that the sprint review and retrospective were easy to ground because the PR chain (#75, #76, #77) cleanly maps the decision, the implementation, and the closeout.
- Wants the sprint-start workflow to verify remote sprint-branch presence before the first item branch opens, so PR creation does not discover that gap late.

## Team Improvements

1. **Push the sprint branch to `origin` before the first item PR**: Sprint 19 discovered the remote sprint-branch gap only when opening the first item PR. The sprint-start protocol should explicitly verify that the agreed sprint branch exists on `origin` before item execution begins.

2. **Do not mark an item `done` before durable review evidence exists**: Sprint 19 briefly moved Item 65 to `done` before the PR/review record existed. The closeout protocol should explicitly check PR/review evidence before backlog status changes, not after.

3. **Keep role-based sub-agent usage selective but explicit**: Sprint 19 benefited from role-specific sub-agents for PO/SM/reflection work without fragmenting the actual implementation slice. This validates selective delegation rather than blanket delegation.

## Product Improvements

1. **Treat `nodeCi()` as proof of a narrow-helper path, not as a preset-framework commitment**: Sprint 19 proved one repeated Node CI pattern. Future recipe expansion should still require fresh evidence and a separate scope decision.

2. **Preserve discovery-first slicing for ambiguous product themes**: Sprint 19 worked because the team did not force design and delivery into one opaque backlog item. When backlog items contain unresolved API-boundary questions, the project should keep splitting them into discovery and implementation phases.

## Follow-Up Documentation

### Scrum Master Owned

- New BOARD item: Add a sprint-start check requiring the sprint branch to exist on `origin` before the first item PR is opened.
- New BOARD item: Add an item-closeout check requiring durable PR/review evidence before a sprint backlog item can be marked `done`.

### Product Owner Owned

- Record that future recipe-style expansion should treat Sprint 19's `nodeCi()` helper as evidence for the narrow-helper path, not as implicit approval for a broader preset framework.
- Keep the active backlog order unchanged after Sprint 19.
