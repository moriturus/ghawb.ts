# Sprint Retrospective: Sprint 20

## Summary

Sprint 20 delivered all committed work totaling 18 SP: `Ad hoc A1`, Items 67, 68, 69, 70a, 70b, 71, and 72. The sprint materially expanded the repository in three directions: CLI ergonomics, syntax/product-surface coverage, and opt-in authoring ergonomics through composite actions and typed action wrappers.

The sprint also exposed one serious process failure: sprint closeout PR #86 was merged even though hosted CI had already failed. That repository defect was repaired immediately in PR #87, which is now merged and green, so the operational impact is closed. The retrospective focus is therefore on preventing the same closeout failure mode from recurring.

Evidence provenance: merged Sprint 20 item PRs #78 through #85, sprint closeout PR #86, CI repair PR #87, the recorded Sprint 20 review note, and the current `main` snapshot after branch cleanup.

## Role Reflections

### Product Owner — Aoi Sakamoto

- Role-based viewpoint: Aoi represents scope control, acceptance judgment, and product-priority decisions across the sprint.
- Observed that Sprint 20 used backlog splitting well. Item 70 was split into discovery (`70a`) and implementation (`70b`), which kept the package-boundary and emission-path decision out of the implementation slice.
- Observed that the delivered surface was coherent rather than scattered: CLI ergonomics improved, the remaining trigger gap was tightened, composite action support gained a minimal but explicit boundary, and typed-action wrappers became materially more useful.
- Observed that the feature scope itself did not need reprioritization after review; Item 73 remains the correct next backlog target.
- Wants future sprints to preserve the same discovery-first behavior when package boundaries or ownership rules are still ambiguous.

### Scrum Master — Ren Takahashi

- Role-based viewpoint: Ren represents delivery flow, branch/PR discipline, impediment management, and sprint-closeout process quality.
- Observed that the sprint mostly respected backlog order and single-item-at-a-time execution, which kept the implementation easy to audit across a relatively broad Sprint 20 scope.
- Observed the most important failure of the sprint at closeout: hosted CI failure on PR #86 was not treated as a stop condition, and local green checks were incorrectly allowed to substitute for GitHub Actions status.
- Observed a smaller hygiene gap after merge: merged item branches were not cleaned up promptly, and one Item 68 branch remained because its tip had moved after PR merge.
- Wants the closeout checklist to require explicit hosted-CI confirmation before merge and the post-closeout checklist to require local/remote branch cleanup.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the architecture and package-boundary decisions made during the sprint.
- Observed that Sprint 20 handled architectural ambiguity well when it was acknowledged early. ADR 0004 prevented composite-action support from leaking into `@ghawb/sdk` and preserved the explicit package boundary.
- Observed that the sprint successfully expanded convenience surfaces without weakening the core AST ownership rules: CLI inference stayed tightly scoped, typed wrappers remained opt-in, and composite actions stayed outside the workflow SDK.
- Observed that the closeout CI failure was not caused by an architecture error so much as a release-process gap: the repository added new build surfaces but failed to treat workflow definitions as release-critical artifacts.
- Wants future package-surface additions to carry an explicit "hosted CI workflow touched or revalidated" check whenever build surfaces change.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent verification quality, review rigor, and coverage discipline.
- Observed that the sprint kept its broad scope reviewable because each item still carried targeted tests plus `bun run check`, and the post-closeout CI repair preserved thresholds by adding tests rather than weakening the bar.
- Observed that Sprint 20 proved a blind spot in the team's evidence model: local validation was treated as sufficient proof even when hosted CI is the actual merge gate.
- Observed that the follow-up coverage abort was valuable signal. It confirmed that the repo still cares about the 100% lines/statements/functions contract and did not paper over the problem by relaxing thresholds.
- Wants future closeout evidence to include the hosted check result explicitly, not just the command list run locally.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent repository ergonomics, workflow definitions, branch hygiene, and tooling friction.
- Observed that Sprint 20 improved user-facing tooling meaningfully: short flags, inferred output paths, and typed-action examples all reduce friction for actual repository users.
- Observed that the biggest tooling failure was in the repository's own CI definition. The committed workflows still had stale hard-coded build steps after the package surface changed, which means self-hosted workflow maintenance was not treated as part of the implementation surface strongly enough.
- Observed that branch cleanup after item merge was inconsistent, which made it harder to see which branches still represented active work.
- Wants workflow regeneration / verification and branch cleanup to be treated as standard closeout hygiene rather than optional housekeeping.

## Team Improvements

1. **Require explicit hosted-CI success before sprint closeout merge**: Sprint 20 showed that local green checks are not an acceptable substitute for GitHub-hosted merge-gate status. The closeout protocol should require an explicit hosted status confirmation step immediately before merge.

2. **Add post-closeout branch cleanup to the protocol**: Merged item and sprint branches should be deleted locally and on `origin` once the sprint is closed, with an explicit exception only when a branch still intentionally holds follow-up work.

3. **Treat workflow-definition updates as release-surface changes**: When a sprint adds packages or changes build surfaces, the team should verify that committed workflow sources and generated YAML still reflect the current repository contract before closeout.

## Product Improvements

1. **Keep discovery-first splitting for architecture-adjacent features**: Sprint 20 benefited from separating Item 70 into ADR-backed discovery and a minimal implementation slice. Future composite-action expansion or other package-boundary work should preserve that approach.

2. **Prefer hardening work ahead of broader surface expansion when repository-contract drift is found**: Sprint 20's CI repair came after closeout, but the product lesson is broader. When workflow generation, coverage enforcement, or packaging surfaces drift from reality, hardening work should take priority over new convenience features until the contract is restored.
