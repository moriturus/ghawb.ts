# Sprint Retrospective: Sprint 21

## Summary

Sprint 21 delivered all committed work totaling 9 SP: Items 74, 76, and 73. The sprint intentionally stayed narrow and hardening-first: it refreshed the repository's self-hosting workflow sources, strengthened Deno compatibility evidence, and improved onboarding documentation after the higher-risk hardening work was complete.

Sprint 21 also showed that the team acted on the biggest Sprint 20 process lesson. Item PRs were merged into the sprint branch first, the sprint closeout PR waited for hosted CI, and the sprint closed cleanly on `main`. The main remaining friction was narrower: documentation-heavy work still allowed API-reference drift to survive until late review, and Deno compatibility for `@ghawb/yaml-import` remains less proven than adjacent public entrypoints after a permission-sensitive parser issue was intentionally deferred.

Evidence provenance: Sprint 21 item PRs #88, #89, and #90, sprint closeout PR #91, the Sprint 21 review note, and the current `main` snapshot after sprint closeout.

## Role Reflections

### Product Owner — Aoi Sakamoto

- Role-based viewpoint: Aoi represents scope control, acceptance judgment, and product-priority decisions across the sprint.
- Observed that the hardening-first ordering paid off. Refreshing the repository's own workflow sources and strengthening Deno compatibility before widening onboarding docs kept the sprint coherent and reduced the risk of documenting stale or drifting guidance.
- Observed that underfilling the sprint was the right call. Leaving 11 SP unused was preferable to pulling unresolved package-boundary work into a sprint whose value came from restoring confidence in shipped surfaces.
- Observed that Item 73 still exposed a product-surface weakness: onboarding docs improved, but API_REFERENCE drift was only fully corrected after review feedback rather than during the first documentation pass.
- Wants the next sprint to preserve the standing order of Item 75a then Item 75b, while capturing the newly surfaced `@ghawb/yaml-import` Deno compatibility concern as separate follow-up backlog rather than reopening Sprint 21 scope.

### Scrum Master — Ren Takahashi

- Role-based viewpoint: Ren represents delivery flow, branch and PR discipline, impediment management, and sprint-closeout process quality.
- Observed that Sprint 21 followed the intended execution pattern end to end: one backlog item at a time, item branches from the sprint branch, item PRs into the sprint branch, then one closeout PR into `main`.
- Observed that scoped role-based collaboration was used effectively. Product Owner, Scrum Master, implementation, and review responsibilities stayed within their boundaries without pretending the work was parallel when it was sequential by backlog order.
- Observed that the largest remaining process gap was documentation sync during doc-heavy items. README and Cookbook expansion triggered necessary API reference corrections late, which means review discovered drift that could have been caught by an earlier documentation-consistency pass.
- Wants the item-closeout checklist to treat public-doc changes as a coordinated surface: when onboarding docs move, API reference and contributor-facing guidance should be re-checked before the item is presented as review-ready.

### Developer: SDK / Architecture — Mio Kanda

- Role-based viewpoint: Mio can credibly represent the architecture and package-boundary decisions exercised during Items 74 and 76.
- Observed that Sprint 21 improved the repository's self-hosting proof without broadening the core architecture. Workflow source modernization used already shipped authoring surfaces rather than inventing new APIs for convenience.
- Observed that the Deno coverage slice stayed disciplined: it improved representative public-entrypoint confidence while preserving Bun as the primary integrated authority and Node as the secondary runtime confirmation path.
- Observed one bounded architecture-adjacent risk: attempting to extend the Deno slice to `@ghawb/yaml-import` exposed permission-sensitive behavior in the YAML parsing path, which remains a real follow-up even though it was correctly kept out of Sprint 21 scope.
- Wants future package-boundary work to stay discovery-first and not let the existence of `nodeCi()` or self-hosting examples force premature package churn.

### Developer: Quality / Testing — Haru Nishimura

- Role-based viewpoint: Haru can credibly represent verification quality, review rigor, and cross-runtime test confidence.
- Observed that Sprint 21 kept its evidence chain strong: Item 74 verified workflow synchronization and Node compatibility, Item 76 strengthened Deno tests without weakening Bun or Node checks, and Item 73 finished with full pre-push verification before merge.
- Observed that review quality still added real value late in the sprint. The Item 73 API-reference drift was caught through review rather than through an earlier documentation consistency sweep, which means the review process prevented stale public guidance from shipping.
- Observed that the Deno/yaml-import parser issue is not speculative; it appeared during test expansion work and should be treated as confirmed compatibility debt even though the sprint correctly narrowed scope instead of fixing it ad hoc.
- Wants future documentation-heavy items to include an explicit "public docs consistency" proof step, and future Deno compatibility work to favor representative package-entrypoint coverage plus targeted failure-mode reproduction.

### Developer: Tooling / Workflow — Yui Morita

- Role-based viewpoint: Yui can credibly represent onboarding ergonomics, repository workflow clarity, and documentation usability.
- Observed that Sprint 21 materially improved the first-success path: package-choice guidance is clearer, the `workflows/` source contract is easier to understand, and the cookbook now gives users a practical `@ghawb/yaml-import` entry point.
- Observed that documentation work is more coupled than it first appears. README and Cookbook improvements exposed stale API_REFERENCE details, which made the doc item feel "done" later than the initial visible edits suggested.
- Observed that the sprint's branch and PR flow was easier to follow than Sprint 20 because each item left behind clearer review and acceptance evidence on its own PR before the closeout PR.
- Wants future documentation work to start from an explicit doc-surface map so README, Cookbook, API reference, and contributor guidance are checked together instead of being updated opportunistically.

## Team Improvements

1. **Add a documentation-surface sync check to item closeout for doc-heavy changes**: When README, Cookbook, or other onboarding docs change, the review-ready checklist should also require a quick consistency pass across API reference and contributor-facing docs so contract drift is caught before late review.

2. **Keep the improved Sprint 21 branch and hosted-CI discipline as the normal closeout expectation**: Sprint 21 validated the Sprint 20 process corrections in practice. The standing Scrum Master follow-ups about hosted-CI confirmation, branch cleanup, and workflow-contract sync remain the right closeout checklist priorities until they are fully codified.

## Product Improvements

1. **Capture the `@ghawb/yaml-import` Deno compatibility gap as explicit backlog work**: Sprint 21 found a real permission-sensitive parser issue while attempting to extend Deno coverage. That debt should be tracked as its own backlog item rather than left as tribal knowledge from the sprint.

2. **Preserve discovery-first handling for Item 75 package-boundary work**: Sprint 21 reinforced that hardening and documentation improvements can ship safely without reopening architecture decisions. Item 75a should remain the next step, with Item 75b gated on a fresh recommendation rather than momentum.
