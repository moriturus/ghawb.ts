# Sprint 25 Retrospective

## Scope

Sprint 25 committed Items 84-86 for a total of 10 story points. The sprint aimed to implement the reviewed CLI config-file path, resync public docs with the current product contract, and add automated guardrails for maintained docs/examples.

The committed scope shipped, and the merged branch now exposes a refined CLI contract:

- render-plan manifests are addressed through `--bulk`
- per-target render-time injection is addressed through `--config`
- workflow modules can read injected values through `getRenderConfig<T>()`
- maintained docs/examples are guarded by `bun run verify:docs`

## Role Reflections

### Aoi Sakamoto, Product Owner

- Can credibly claim the original Sprint 25 scope decision and the later judgment that the user-intended config injection behavior should take ownership of `--config`.
- Observed that the product direction became clearer once the render-plan path and render-time injection path stopped sharing the same flag name.
- Wants the next backlog intake to consolidate the now-shipped contract across sprint records and public docs before more CLI feature expansion.

### Ren Takahashi, Scrum Master

- Can credibly represent the sprint closeout flow and post-merge documentation coordination.
- Observed one concrete process gap: the repository's docs guardrails passed even though a later README CLI Usage block still referenced the retired `--config` manifest spelling.
- Wants the docs guardrails to check for cross-section consistency, not only for the presence of a few canonical strings.

### Mio Kanda, Developer

- Can credibly represent the SDK/CLI boundary refinement around `getRenderConfig<T>()`.
- Observed that keeping render-time config injection at the CLI boundary preserved the SDK architecture while still enabling `.onPush(config.onPushBranches)` style authoring.
- Wants future work to keep the render-plan parser and the injected-data parser explicitly separate, especially around TOML scope.

### Haru Nishimura, Developer / Validation

- Can credibly represent the expanded CLI and SDK test coverage.
- Observed that the most important regression was not basic parsing but module re-evaluation when the same `--input` is rendered more than once with different config files.
- Wants future validation work to preserve regression tests for per-target cache isolation and docs-contract drift.

### Yui Morita, Developer / Tooling

- Can credibly represent the docs and CLI ergonomics perspective.
- Observed that the rename from `--bulk-config` to `--bulk` made the CLI easier to explain, but the repository still had one stale README section because the documentation contract was duplicated.
- Wants the maintained CLI usage guidance reduced to fewer duplicated snippets so command-surface renames are cheaper to land safely.

## Team Improvements

1. Expand the docs guardrails so they detect stale renamed flags across every maintained CLI usage section, not just the current canonical snippet.
2. Add a sprint-review consistency check that compares merged CLI contract changes against the sprint backlog wording before the review is recorded as complete.

## Product Improvements

1. Reconcile the remaining README `--config` manifest wording with the shipped `--bulk` / per-target `--config` contract.
2. Make the shipped config model explicit in sprint/backlog records so the repository history reflects what users can actually do on `main`.
3. Keep future CLI config work narrow: clarify TOML injection scope before broadening parser support.
