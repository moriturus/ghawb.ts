# Sprint 26 Retrospective

## Scope

Sprint 26 committed Items 87-90 for a total of 10 story points. The sprint aimed to reconcile the maintained CLI config contract, harden docs guardrails against stale wording, add a narrow job-helper slice, and clarify the quickest package adoption path.

The committed scope shipped:

- README, SPEC, and sprint records now agree on `--bulk` for render-plan manifests, per-target `--config` for injected config, and `getRenderConfig<T>()` for workflow-module access
- docs guardrails now fail when stale `--config` manifest wording reappears
- `@ghawb/job-helpers` now exposes `nodeBootstrap()` as a narrow additive helper
- README package-selection guidance now presents a shorter adoption path

## Role Reflections

### Aoi Sakamoto, Product Owner

- Can credibly claim the backlog-shaping decisions that kept Sprint 26 focused on contract hardening, helper ergonomics, and onboarding clarity.
- Observed that the product direction stayed coherent because the sprint did not reopen the CLI contract split or mix helper work into the SDK core.
- Wants future intake to keep package adoption and helper additions additive, with no pressure to expand the core boundary just because a small helper exists.

### Ren Takahashi, Scrum Master

- Can credibly represent the sprint sequencing and closeout discipline.
- Observed that the existing docs guardrail still needed a retrospective-level reminder: review readiness is only as good as the weakest maintained doc surface, not just the canonical snippet.
- Wants the closeout path to keep comparing review evidence, backlog wording, and maintained docs before the sprint is declared done.

### Mio Kanda, Developer

- Can credibly represent the boundary-preserving implementation of `nodeBootstrap()` and the docs contract cleanup around the CLI config path.
- Observed that the helper stayed well-scoped because it remained additive and package-local.
- Wants future helper work to stay narrow and to avoid any temptation to move convenience APIs back into `@ghawb/sdk`.

### Haru Nishimura, Developer / Validation

- Can credibly represent the verification side of the sprint through the docs and job-helper tests.
- Observed that the new guardrail is only useful if it is paired with an example of the stale wording it is meant to reject.
- Wants future validation work to keep a regression test for the retired CLI wording and a direct test for each additive helper slice.

### Yui Morita, Developer / Tooling

- Can credibly represent the docs and onboarding clarity work.
- Observed that the package-adoption path became easier to skim once the README stopped trying to describe too many starting points at once.
- Wants future docs edits to preserve the current "start here, then add this" shape instead of reintroducing duplicated guidance in multiple sections.

## Team Improvements

1. Keep the sprint-review and retrospective evidence path explicit enough that reviewers can compare maintained docs, backlog wording, and shipped behavior without hunting through multiple files.
2. Continue to treat docs guardrails as a cross-section consistency problem, not just a canonical-snippet problem, whenever a user-facing flag or contract wording changes.

## Product Improvements

1. Preserve the additive package boundary for helper ergonomics work; `@ghawb/job-helpers` should keep absorbing narrow repeats only when they clearly shorten an authoring path.
2. Keep the package-adoption path short and ordered in README so the next package addition stays obvious to first-time users.
