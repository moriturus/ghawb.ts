# Sprint Review: Sprint 11

## Summary

Sprint 11 reviewed the backlog slice `Item 33`, `Item 35`, `Item 30`, and `Item 29` (12 out of 15 capacity story points, with a deliberate 3 SP buffer). This was the final sprint — the product backlog is now empty.

- Step-based jobs now support optional `container` and `services` fields with full validation, clone/freeze semantics, and deterministic canonical rendering.
- ~70 validation error messages across all SDK surfaces were enriched with expected formats and valid values following the `[scope] [field] [constraint]. Expected: [format/values]` convention.
- Six Scrum Master BOARD items were codified into concrete PLAYBOOK.md rules covering closeout waiting behavior, evidence provenance, verification gates, external proof planning, no-false-stop execution, and sprint-document synchronization.
- Self-hosted CI workflow expanded to exercise `permissions` and `concurrency`. Package distribution prepared with JSR workspace configuration, CLI `jsr.json`, and SPEC.md distribution contract.

## Increment Demo

### 1. Container and services rendering (Item 33)

```sh
bun -e "
import { defineWorkflow, createWorkflowId, createJobId, createWorkflowRenderPayload } from './packages/sdk/src/index.ts';
const w = defineWorkflow({ id: createWorkflowId('demo'), name: 'Demo' })
  .onPush()
  .addJob(createJobId('test'), (job) => {
    job.runsOn('ubuntu-latest')
      .container({ image: 'node:20', ports: [3000], env: { NODE_ENV: 'test' } })
      .services({ db: { image: 'postgres:15', ports: [5432], env: { POSTGRES_PASSWORD: 'test' } } })
      .run('npm test');
  }).build();
console.log(JSON.stringify(createWorkflowRenderPayload(w).jobs, null, 2));
"
```

Output confirms canonical field ordering: `runs-on` → `container` → `services` → `steps`, with container internal order `image` → `env` → `ports`.

### 2. Enriched validation diagnostics (Item 35)

```sh
bun -e "
import { defineWorkflow, createWorkflowId, createJobId } from './packages/sdk/src/index.ts';
try {
  defineWorkflow({ id: createWorkflowId('bad'), name: 'Bad' })
    .onPush()
    .addJob(createJobId('j'), (job) => {
      job.runsOn('ubuntu-latest')
        .container({ image: '' })
        .services({ '123bad': { image: 'x' } })
        .run('echo');
    }).build();
} catch (e) { e.issues.forEach(i => console.log(i)); }
"
```

Output:

```
job "j" container image must be a non-blank string
job "j" service "123bad" name must match ^[a-zA-Z_][a-zA-Z0-9_-]*$. Expected: a letter or underscore start, followed by letters, digits, underscores, or hyphens
```

Enriched messages preserve the existing prefix structure while appending actionable guidance.

## Findings

### Confirmed: no material issues

- Container/services implementation follows the same explicit-boundary pattern as all other optional job fields. Validation, cloning, freezing, and rendering are complete.
- Enriched error messages preserve programmatic prefix structure — no breaking changes to `error.issues` consumption patterns.
- PLAYBOOK.md additions are consistent with existing heading structure and numbering.
- All JSR and npm export paths resolve to existing source modules.
- SPEC.md is synchronized with the delivered implementation across all 4 items.

### Residual note

- The `renderWorkflow()` function requires an explicit emitter argument (no default YAML emitter). This is by-design per ADR 0001 but means the SDK alone cannot produce YAML output — the CLI package or user-supplied emitter is required. This is documented and not a gap.
