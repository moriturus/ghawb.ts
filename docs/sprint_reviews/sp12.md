# Sprint Review: Sprint 12

## Summary

Sprint 12 reviewed the backlog slice `Item 36`, `Item 37`, `Item 38`, `Item 39`, `Item 40`, and `Item 44` (18 out of 20 capacity story points, with a deliberate 2 SP buffer). Items 36–40 form the "syntax breadth" group; Item 44 starts the "type-safety" group.

- Workflow `run-name` and job `name` display fields are now supported with canonical field ordering at both workflow and job level.
- Job `environment` supports both string form (name only) and object form (`name` + optional `url`), with reusable-workflow rejection at build time.
- `pull_request_target` trigger reuses the `pull_request` filter infrastructure with branch/path/types filters and tags rejection.
- `workflow_run` trigger supports required `workflows` array, optional `types` allowlist, and optional `branches`/`branches-ignore` with mutual exclusion.
- 23 simple event triggers are now supported — types-only events validate against per-event allowlists, bare events render as event keys, and `repository_dispatch` accepts arbitrary client-defined type strings.
- `RunnerLabel` constants provide typed autocomplete for 16 known GitHub-hosted runner labels while preserving full backward compatibility with raw string usage.

- This review was performed on `main` after Sprint 12 was merged through sprint closeout PR [#43](https://github.com/moriturus/ghawb.ts/pull/43).

## Increment Demo

### 1. Display name fields — `run-name` and job `name` (Item 36)

```sh
bun -e "
import { defineWorkflow, createWorkflowId, createJobId, createWorkflowRenderPayload } from './packages/sdk/src/index.ts';
const w = defineWorkflow({ id: createWorkflowId('demo'), name: 'Demo' })
  .runName('Deploy \${{ github.sha }}')
  .onPush()
  .addJob(createJobId('build'), (job) => {
    job.displayName('Build & Test').runsOn('ubuntu-latest').run('npm test');
  }).build();
const p = createWorkflowRenderPayload(w);
console.log('run-name:', p['run-name']);
console.log('job name:', p.jobs.build.name);
console.log('workflow fields:', Object.keys(p).join(', '));
console.log('job fields:', Object.keys(p.jobs.build).join(', '));
"
```

Output:

```
run-name: Deploy ${{ github.sha }}
job name: Build & Test
workflow fields: name, run-name, on, jobs
job fields: name, runs-on, steps
```

Confirms `run-name` renders after `name` at workflow level and job `name` renders as the first field in the job payload.

### 5. Simple event triggers (Item 40)

```sh
bun -e "
import { defineWorkflow, createWorkflowId, createJobId, createWorkflowRenderPayload } from './packages/sdk/src/index.ts';
const w = defineWorkflow({ id: createWorkflowId('se'), name: 'SE' })
  .onEvent('issues', { types: ['opened', 'labeled'] })
  .onEvent('release', { types: ['published'] })
  .onEvent('fork')
  .onEvent('repository_dispatch', { types: ['deploy', 'rollback'] })
  .onPush()
  .addJob(createJobId('j'), (job) => {
    job.runsOn('ubuntu-latest').run('echo ok');
  }).build();
const p = createWorkflowRenderPayload(w);
console.log('trigger keys:', Object.keys(p.on).join(', '));
"
```

Output:

```
trigger keys: push, fork, issues, release, repository_dispatch
```

Confirms canonical trigger ordering: complex triggers (`push`) first, then simple events in alphabetical order (`fork`, `issues`, `release`, `repository_dispatch`). Types-only events validate against per-event allowlists. Bare events (`fork`) render as event keys with `null` value. `repository_dispatch` accepts arbitrary type strings.

## Findings

### Confirmed: no material issues

- All six items follow the established model/builder/validation/rendering/conformance pattern. No architectural novelty was introduced.
- Canonical field orders at workflow, job, and trigger levels are consistent between SPEC.md, render payload output, and conformance fixtures.
- Validation coverage is thorough: blank values, unsupported filters, duplicate triggers, mutual exclusion rules, and reusable-workflow boundary rejections are all tested with dedicated conformance fixtures.
- All 23 simple event triggers render correctly in isolation and combination with existing triggers.
- `RunnerLabel` is purely additive — existing `string` usage is unaffected.
- SPEC.md is synchronized with the delivered implementation across all 6 items.
- Sprint 12 delivered 18 SP, a 50% increase over the 12 SP delivered in Sprints 10–11. The velocity increase was successful because all items followed well-established implementation patterns with no architectural novelty.

### Residual note

- The `RunnerLabel` set is a snapshot of GitHub-hosted runners at implementation time. New runner variants (e.g., ARM runners, future OS versions) will require explicit additions. This is by-design per Item 44's acceptance criteria — unknown labels are accepted silently via the string fallback, so there is no runtime breakage when GitHub adds new runners.
