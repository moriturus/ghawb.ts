# Sprint Review: Sprint 14

## Summary

Sprint 14 delivered all three committed items (12 SP): Item 47 (run script references, 4 SP), Item 43 (README rewrite, 3 SP), and Item 45 (release automation, 5 SP). The sprint closed with PR #51 from `sprint-14` into `main`.

- Item 47 added `runScript()` to the SDK, supporting local file references, shell-prefixed file references, and an optional expand mode that inlines the script body.
- Item 43 rewrote the README for external consumers with a value proposition, install instructions, five verified examples, supported/unsupported feature summary, and `actionlint` guidance.
- Item 45 adopted Changesets for lockstep versioning and added self-hosted release and publish workflows using the SDK.

Review note about repository state:

- This review was performed after Sprint 14 had already been merged through sprint closeout PR #51.
- A post-merge CI fix was required for a `tsc` strict compilation error in `runScript()` — the fix was pushed to the sprint branch and CI passed before merge.
- Evidence came from the merged increment, the sprint backlog, the specification, and verification output captured during delivery.

## Increment Demo

### 1. Run script reference surface (Item 47)

The merged increment now includes:

- `runScript({ path, shell?, expand? })` builder method on `JobBuilder`
- `ScriptReference` model type with path, shell, and expand fields
- Reference mode: emits `shell path` or just `path` as the `run` value
- Expand mode: reads the script file at build time and emits its contents as the `run` value
- Validation: blank path, blank shell, unreadable file (expand mode), and double shell specification all produce clear errors
- 13 builder tests, 6 renderer tests, 4 conformance fixtures

### 2. README rewrite (Item 43)

The README now includes:

- Value proposition and feature summary for external consumers
- Install instructions for npm, pnpm, yarn, and bun
- Five verified examples: hero CI, CI with concurrency, deployment with environment, matrix build, and reusable workflow
- Supported/unsupported feature matrix
- `actionlint` complementary usage guidance
- Precise "100% SDK line coverage" scope clarification
- Runtime compatibility: Node 24+, Bun 1.x, Deno 2.x

All five examples are verified by `tests/node/readme-examples.test.ts` to compile and render valid YAML.

### 3. Release automation (Item 45)

The repository now includes:

- `.changeset/config.json` with lockstep versioning across `@ghawb/sdk`, `@ghawb/shared`, `@ghawb/cli`
- `workflows/release.ts`: self-hosted release PR workflow using the SDK
- `workflows/publish.ts`: self-hosted tag-triggered publish workflow (npm + JSR + GitHub Release)
- `.github/workflows/release.yml` and `.github/workflows/publish.yml`: generated YAML
- `RELEASING.md`: complete release flow documentation
- Dry-run proof: all three packages pass `npm publish --dry-run`

### 4. Delivery verification

Sprint delivery verification completed successfully:

- 354 Bun/Node tests passed (up from 323 in Sprint 13)
- 75 Deno tests passed (up from 71 in Sprint 13)
- 100% line, function, and statement coverage maintained
- format, lint, and typecheck all passed

## Findings

### CI failure: Bun-vs-tsc type checking gap

- `runScript()` passed single strings to `WorkflowValidationError(...)` which expects `readonly string[]`.
- Bun's runtime TypeScript execution accepted this, but `tsc` 5.9.3 strict mode rejected it.
- Fixed by wrapping error messages in arrays and removing now-unreachable build-time validation.
- Lesson recorded in `docs/LEARN.md`.

### New backlog items surfaced

Two new product backlog items were identified during Sprint 14:

- **Item 48** (3 SP): Reusable workflow object injection — allow `usesWorkflow()` to accept `WorkflowBuilder` or `WorkflowDefinition` directly, with `workflow_call` validation at the caller's `build()` time.
- **Item 49** (2 SP): Fix `npm install`/`npm ci` failure — the `workspace:*` protocol in `package.json` is not supported by npm.

## Product Owner Decision — Aoi Sakamoto

- Sprint 14 delivered all committed scope. No reprioritization is needed for this sprint.
- Item 48 and Item 49 are accepted into the product backlog for future sprint selection.
- Item 49 (npm install fix) should be prioritized ahead of Item 48 since it blocks npm consumers from developing locally.
