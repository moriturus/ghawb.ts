## Increment Demo

### 1. Expression helper MVP (Item 51)

Expression helpers eliminate raw `${{ }}` string interpolation for common patterns:

```typescript
import { expr, github, always, env, secrets, steps } from "@ghawb/sdk";

// Before: job.run("npm test", { if: "${{ always() }}" });
// After:
job.run("npm test", { if: expr(always()) });
// → if: "${{ always() }}"

expr(github("ref"));          // → "${{ github.ref }}"
expr(env("NODE_ENV"));        // → "${{ env.NODE_ENV }}"
expr(secrets("TOKEN"));       // → "${{ secrets.TOKEN }}"
expr(steps("build").outputs("result")); // → "${{ steps.build.outputs.result }}"
```

Helpers: `expr()`, `github()`, `env()`, `secrets()`, `steps()`, `matrix()`, `inputs()`, `success()`, `always()`, `cancelled()`, `failure()`. Build-time validation rejects empty/blank expressions. Semantic evaluation is an explicit non-goal.

Verified: 52 expression-specific tests, conformance fixture updated. All existing raw-string paths remain backward compatible.

### 2. Advanced `runs-on` support (Item 52)

Step-based jobs now support the GitHub Actions object form of `runs-on`:

```typescript
job.runsOn({ group: "ubuntu-runners", labels: ["self-hosted", "linux"] });
// renders → runs-on: { group: ubuntu-runners, labels: [self-hosted, linux] }
```

All three combinations supported: `group`-only, `labels`-only, and `group`+`labels`. At least one field required. Object form and string/array form are mutually exclusive. Unsupported shapes fail at build time.

Verified: 7 builder tests, 3 renderer tests, 1 conformance fixture (3 job variants). `docs/SYNTAX_COVERAGE.md` updated.

Implementation note: Deno's stricter type narrowing required an explicit `isRunsOnObject()` type guard instead of inline `typeof`/`Array.isArray` checks for the `RunsOnTarget` union.

### 3. actionlint bridge (Item 53)

New CLI command: `ghawb lint <files...>`

```
$ ghawb lint ci.yml deploy.yml
# Runs actionlint against the specified files

$ ghawb lint ci.yml   # when actionlint is not installed
Error: actionlint is not installed. ...
Install: https://github.com/rhysd/actionlint
Exit code: 1
```

When `actionlint` is not found on `PATH`, the CLI exits non-zero with a clear message naming the tool and linking to installation instructions. No silent skip or zero-exit fallback.

Verified: 5 unit tests using DI-mocked dependencies (`findExecutable`, `runCommand`). Architecture boundary preserved — no YAML parsing added to core SDK.

### 4. API reference and cookbook docs (Item 54)

Two new documentation files:

- `docs/API_REFERENCE.md` (325 lines) — Full SDK public API surface organized by concern: workflow builder, job builder, step builder, expressions, renderer, validation, model types, and known gotchas (including the `displayName()` naming decision per `docs/LEARN.md`).
- `docs/COOKBOOK.md` (374 lines) — 11 practical workflow recipes: basic CI, matrix builds, reusable workflows, cron schedules, environment deployments, concurrency control, container jobs, and more.

`docs/INDEX.md` updated with links to both new files.

### 5. Opt-in YAML import package (Item 55)

New package: `@ghawb/yaml-import`

```typescript
import { importReusableWorkflow } from "@ghawb/yaml-import";

const ref = await importReusableWorkflow("./shared-ci.yml");
// → "./.github/workflows/shared-ci.yml" (WorkflowRef)

// Pass directly to usesWorkflow():
job.usesWorkflow(ref, { with: { "node-version": "20" } });
```

Handles all three `workflow_call` trigger forms (object, bare string, list). Rejects workflows without `workflow_call` trigger with `YamlImportError` diagnostics. YAML parsing stays in the opt-in package; `@ghawb/sdk` remains YAML-input-free per ADR 0001.

Verified: 13 tests covering happy paths and all error cases. DI pattern (`readFile` injection) for testability. Package configs: `package.json`, `tsconfig.build.json`, `jsr.json`, `deno.json`. Root configs updated: vitest, tsconfig paths, deno imports, jsr workspace.

## Findings

### No material bugs or contract gaps

The Sprint 16 increment is clean. All five items pass their acceptance criteria and definition of done. No validation holes, contract ambiguities, or untested paths were identified during review.

### Observation: Deno type-narrowing strictness (non-blocking, documented)

Item 52 revealed that Deno's type-checker is stricter than Bun/Node TypeScript for inline union narrowing with `typeof`/`Array.isArray`. The workaround (explicit type guard function) is clean and already documented in `docs/LEARN.md`. This is a known cross-runtime characteristic, not a bug.

### Observation: Package count growth

The monorepo now contains four packages (`@ghawb/sdk`, `@ghawb/shared`, `@ghawb/cli`, `@ghawb/yaml-import`). The root configuration surface (vitest, tsconfig, deno.json, jsr.json, package.json) must be updated in five places when adding a new package. This is manageable at the current scale but worth noting if additional packages are planned.

### Observation: Product backlog exhaustion (continued)

The product backlog remains empty after Sprint 16. This is the second consecutive sprint where backlog exhaustion was noted. Future work requires new intake from product discovery, user feedback, or technical debt identification.

## Product Owner Decision — Aoi Sakamoto

- Sprint 16 delivered all committed scope. No reprioritization is needed.
- All five items satisfied their Definition of Done and Acceptance Criteria. Item-level PO acceptance was performed during sprint execution (each item received explicit acceptance before merge via PR comments on PRs #59–#63).
- The product backlog remains empty. No items remain for sprint selection.
- The increment is release-worthy. Sprint 16 added significant SDK value (expressions, runs-on expansion), developer tooling (actionlint bridge), adoption documentation (API reference, cookbook), and migration tooling (YAML import). Combined with the release infrastructure from Sprint 14 and the bug fixes from Sprint 15, the project is in a strong position for a 0.1.0 release.
- Next steps should focus on:
  - **Release evaluation**: The accumulated increment across 16 sprints represents a usable, documented, tested SDK. A 0.1.0 release decision should be made before adding more features.
  - **User feedback intake**: The API reference and cookbook provide enough documentation surface to solicit real-world usage feedback.
  - **Backlog replenishment**: If continuing development, discovery should focus on composite action support, additional expression operators, workflow_dispatch UI enhancements, or broader trigger coverage based on user demand signals.
