# Sprint Review: Sprint 17

Sprint 17 delivered all five committed items (8 SP): Item 57 (fix formatting, 1 SP), Item 58 (coverage thresholds, 2 SP), Item 59 (documentation accuracy audit, 2 SP), Item 60 (LEARN.md date integrity audit, 1 SP), and Item 56 (string shorthand for step name, 2 SP). ### 5. String shorthand for step name (Item 56)

`uses()`, `run()`, and `runScript()` now accept a plain string as the second parameter, interpreted as `{ name: value }`:

```typescript
// Before (still works)
job.uses("actions/checkout@v4", { name: "Checkout code" })
job.run("npm test", { name: "Run tests" })

// After (new shorthand)
job.uses("actions/checkout@v4", "Checkout code")
job.run("npm test", "Run tests")
job.runScript({ path: "./deploy.sh" }, "Deploy")
```

Rendered output (verified live):
```json
{
  "steps": [
    { "name": "Checkout code", "uses": "actions/checkout@v4" },
    { "name": "Install deps", "run": "npm install" },
    { "name": "Run tests", "run": "npm test" }
  ]
}
```

Existing `StepMetadata` object usage is fully backward compatible. Empty strings are rejected by existing validation. The backlog originally scoped `uses()` and `run()` only; `runScript()` was evaluated during implementation and included since it carries the same metadata parameter pattern.

`docs/SPEC.md` updated with the string shorthand documentation.

## Findings

### No material bugs or contract gaps

The Sprint 17 increment is clean. All five items pass their acceptance criteria and definition of done. No validation holes, contract ambiguities, or untested paths were identified during review.

### Observation: Item 57 was already resolved on the sprint base

The external project review identified formatting failures on `main`, but Sprint 16's final commit (`93afaa0`) resolved them before the sprint-17 branch was created. Item 57 was correctly accepted without code changes. This is not a planning error — the gap was real when the discovery was conducted, and the item correctly identified the problem. The fix simply arrived through Sprint 16's closing work rather than Sprint 17 execution.

### Observation: Item 60 scope expanded during execution

The sprint backlog identified 3 future-dated LEARN.md entries (Sprint 8 and Sprint 11 lessons). The audit found 2 additional future-dated entries (Sprint 11 temporary files, Sprint 14 Bun runtime). All 5 were corrected. The expanded scope was within the item's spirit and acceptance criteria ("no entry contradicts the commit history"), and the additional work was trivial.

### Observation: Release readiness assessment

Sprint 17 closes the quality-gate and documentation-accuracy gaps identified in the external project review (`gpt/project_review.md`). Combined with the increment accumulated across Sprints 1–16, the project now has:

- Clean quality gates (`bun run check` green)
- Comprehensive coverage enforcement (4 metrics)
- Accurate cross-document feature status
- Trustworthy documentation dates
- Developer-friendly API ergonomics (string shorthand)

The product is in a strong position for a 0.1.0 release evaluation.

## Product Owner Decision — Aoi Sakamoto

- Sprint 17 delivered all committed scope (8/8 SP, 5/5 items). No reprioritization is needed.
- All five items satisfied their Definition of Done and Acceptance Criteria. Item-level PO acceptance was performed during sprint execution (each item received explicit acceptance before merge).
- **Release readiness:** Sprint 17 was designed as a release-prerequisite sprint, and it achieved its goal. The quality-gate, coverage, documentation, and date-integrity gaps from the project review are closed. The 0.1.0 release decision should be made.
- **Backlog state:** Five feature items remain in the product backlog (Items 61–65: typed action wrappers, CLI render+lint, reusable workflow caller-side outputs, expression helper expansion, job presets). These were intentionally deferred to post-release sprints.
- **Priority adjustment for Sprint 18:** No priority changes to the current backlog order. Items 61–65 should be evaluated for Sprint 18 selection after the 0.1.0 release decision, with typed action wrappers (Item 61) remaining the highest-priority feature item.
- **Process observation:** The sprint demonstrated efficient release-readiness cleanup. The lightweight nature of Items 57 and 60 (1 SP each) confirmed that quality-gate and documentation work can be efficiently handled in a single sprint when scoped tightly.
