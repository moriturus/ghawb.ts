# Sprint Review: Sprint 19

## Summary

Sprint 19 delivered both committed items (5 SP): Item 65 (job preset / recipe discovery, 2 SP) and Item 66 (job preset / recipe initial implementation slice, 3 SP). Item PRs #75 and #76 were merged into `sprint-19`, and the sprint closeout PR #77 (`sprint-19` → `main`) was merged on 2026-04-04.

Sprint 19 was a deliberately narrow discovery-plus-delivery sprint. It resolved the open product-direction question around repeated job patterns, rejected a broad preset framework for now, and delivered one additive `@ghawb/sdk` helper that reduces boilerplate for the common Node CI step sequence.

The active product backlog remains Items 67–70. No Sprint 19 item carried over.

## Increment Demo

### 1. Discovery output and package-boundary decision (Item 65)

Evidence:

- [ADR 0002](../adrs/0002-scope-job-recipes-to-a-narrow-node-ci-helper.md) records the three-way comparison between cookbook-only guidance, a narrow helper API, and a broader preset / recipe layer.
- [Sprint 19 backlog](../sprint_backlogs/sp19.md) records the resulting implementation boundary for Item 66.

Result:

- Discovery closed the major open design questions for the sprint-safe slice.
- The selected boundary is `@ghawb/sdk`.
- The chosen implementation target is a single additive Node CI helper rather than a new preset subsystem.

### 2. Node CI helper demo (Item 66)

Command:

```sh
bun -e 'import { createJobId, createWorkflowId, defineWorkflow, createWorkflowRenderPayload } from "./packages/sdk/src/index.ts"; const workflow = defineWorkflow({ id: createWorkflowId("demo_ci"), name: "Demo CI" }).onPush({ branches: ["main"] }).addJob(createJobId("test"), (job) => { job.runsOn("ubuntu-latest").nodeCi({ nodeVersion: "22", cache: "pnpm", install: "pnpm install --frozen-lockfile", test: "pnpm test" }); }).build(); const payload = createWorkflowRenderPayload(workflow); console.log(JSON.stringify(payload.jobs.test.steps, null, 2));'
```

Result:

- The rendered step sequence contains exactly four additive steps: `Checkout`, `Setup Node`, `Install`, and `Test`.
- `Setup Node` renders `actions/setup-node@v6` with the expected `node-version` and cache metadata.
- `Install` and `Test` remain explicit commands, so users can still customize commands without adopting a broader recipe DSL.

### 3. Full-sprint verification


Result:

- The worktree is clean on `main`.

## Findings

### No material bugs or contract gaps

Sprint 19 meets the committed backlog intent. The discovery decision, implemented API, validation behavior, tests, and documentation all align with the Sprint 19 backlog and the current specification. No confirmed bug, missing acceptance slice, or undocumented contract drift was identified during review.

### Observation: the increment stayed inside the sprint-safe boundary

The delivered helper is intentionally narrow. `JobBuilder.nodeCi()` appends a repeated Node CI step sequence but does not expand into workflow presets, trigger defaults, permission presets, or multi-language abstractions. That matches ADR 0002 and keeps the new surface additive rather than committing the project to a premature recipe framework.

### Observation: documentation and validation moved together

Sprint 19 changed the public authoring path, but the README, Cookbook, API reference, sprint backlog, and specification were updated in the same slice. The helper also ships with validation for blank `nodeVersion`, `install`, and `test` values, with explicit tests covering those failure modes.

### Residual risk: broader preset families remain unproven

Sprint 19 proves the first narrow helper slice, not a general preset architecture. If future backlog work revisits Bun CI, pnpm-specific helpers, or broader workflow recipes, that work should still be treated as fresh scope rather than implied by `nodeCi()`.

## Product Owner Decision — Aoi Sakamoto

- Sprint 19 delivered all committed scope (5/5 SP, 2/2 items). No item requires rework or carry-over.
- No active-backlog reprioritization is needed. The next backlog order remains Item 67 → Item 68 → Item 69 → Item 70.
- The Sprint 19 increment validates the product preference for additive explicit APIs over a broader preset abstraction. Future recipe work should require fresh evidence rather than assuming a framework direction.
- The standing release judgment remains unchanged: this sprint does not introduce a `0.1.0` release decision.
- Sprint execution followed the repository workflow expectations: item PRs were merged into the sprint branch first, the sprint closeout PR was merged after hosted CI passed, and item-level Product Owner acceptance happened during execution rather than being deferred to this review.
