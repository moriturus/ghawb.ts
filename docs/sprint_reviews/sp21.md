# Sprint Review: Sprint 21

## Summary

Sprint 21 delivered all committed backlog items (Items 74, 76, and 73) for a total of 9/9 committed story points. Item PRs #88, #89, and #90 were merged into `sprint-21`, and the sprint closeout PR #91 (`sprint-21` -> `main`) was merged on 2026-04-05.

The increment focused on hardening and clarity rather than new broad surface area: committed self-hosting workflow sources now use current authoring patterns, Deno compatibility evidence moved beyond the prior smoke-oriented baseline, and onboarding docs now explain the package boundaries and reusable-workflow paths more clearly.

No committed sprint item carried over. The active product backlog remains Item 75a followed by Item 75b.

## Findings

### No material bugs or contract gaps identified in the delivered sprint increment

The delivered workflow sources, Deno test contract, onboarding docs, sprint backlog record, and current specification align with the Sprint 21 backlog. No missing acceptance slice, feature carry-over, or confirmed implementation defect was identified during review.

### Observation: the workflow rewrite improved the repository's self-hosting proof without behavior drift

Sprint 21 updated the committed `workflows/*.ts` modules to current authoring surfaces, but the generated workflow behavior remained stable. The repository now presents stronger in-repo examples for `RunnerLabel` usage and typed action wrappers without introducing a new authoring abstraction.

### Observation: Deno compatibility evidence is stronger, but still intentionally bounded

Sprint 21 materially improved Deno confidence by adding representative public-entrypoint checks. That said, the repository still treats Bun as the primary integrated quality authority and Node as the secondary runtime confirmation path. This remains an intentional product boundary, not an incomplete migration.

### Residual risk: package-boundary discovery for high-level helpers remains the next open decision

Sprint 21 intentionally deferred the `nodeCi()` package-boundary question. The remaining product risk is architectural rather than operational: Item 75a still needs to decide whether the current `@ghawb/sdk` placement should remain or move behind a different opt-in boundary before any follow-up implementation begins.

### Observation: sprint execution discipline improved relative to Sprint 20

Sprint 21 followed the intended repository flow end to end: item PRs were merged into the sprint branch first, the sprint closeout PR was opened from `sprint-21` into `main`, and hosted CI succeeded before closeout merge. That removes the Sprint 20 review finding around merging closeout before hosted confirmation.

## Product Owner Decision — Aoi Sakamoto

- Sprint 21 delivered all committed scope (9/9 SP, 3/3 items). No item requires rework or carry-over.
- No active-backlog reprioritization is needed. The next backlog order remains Item 75a followed by Item 75b.
- The hardening-first Sprint 20 reprioritization is validated by the delivered repository state: self-hosting examples are clearer, Deno compatibility evidence is stronger, and onboarding docs are more usable without broadening the architecture prematurely.
- The package-boundary question for high-level helpers remains discovery-first scope. Item 75b must not begin unless Item 75a explicitly justifies migration away from the current `@ghawb/sdk` placement.
- Sprint execution used the repository's intended branch, PR, review, and hosted-CI closeout gates successfully; that closeout discipline should remain the expected standard for future sprints.
