# Sprint Review: Sprint 25

## Summary

Sprint 25 delivered all committed backlog items for a total of 10/10 committed story points. Item PRs [#103](https://github.com/moriturus/ghawb.ts/pull/103), [#104](https://github.com/moriturus/ghawb.ts/pull/104), and [#105](https://github.com/moriturus/ghawb.ts/pull/105) were merged into `sprint-25`, and the sprint closeout PR [#106](https://github.com/moriturus/ghawb.ts/pull/106) was merged into `main` on 2026-04-06.

The delivered increment proved three things:

- the CLI can render through a file-owned multi-target plan path and now also supports per-target render-time config injection
- the public docs and specification were resynchronized with the current repository contract
- maintained docs/examples now have an automated drift guardrail

No committed sprint item carried over. However, the final merged increment now exposes a narrower and clearer CLI contract than the original Sprint 25 backlog wording: render-plan manifests are addressed through `--bulk`, while `--config` is bound to the immediately preceding `--input` and is exposed inside workflow modules through `getRenderConfig<T>()`.

## Findings

### Confirmed issue: sprint records no longer fully match the shipped CLI contract

The current implementation and specification now define:

- `ghawb render --bulk <file>` for render-plan manifests
- `ghawb render --input <module.ts> --config <data-file>` for per-target render-time injection
- SDK access through `getRenderConfig<T>()`

Direct evidence:

- [README.md](../../README.md)
- [docs/SPEC.md](../SPEC.md)
- [packages/cli/src/index.ts](../../packages/cli/src/index.ts)
- [packages/sdk/src/index.ts](../../packages/sdk/src/index.ts)

But the original Sprint 25 backlog still describes Item 84 as the implementation of a `config-manifest` path and does not mention `--bulk`, `getRenderConfig()`, or the repurposed `--config` semantics. That means the committed sprint scope was delivered, but the sprint record no longer describes the exact contract that shipped on `main`.

### Confirmed issue: one maintained README section still contains the old `--config` manifest wording

The top-level README was updated for the new contract, but the later CLI Usage section still says:

- `ghawb render --config ghawb.render.json`
- `render` accepts a CLI-owned config manifest through `--config <file>`

Direct evidence:

- [README.md](/Volumes/ExSSD/gits/oss/ghawb.ts/README.md#L331)

This is a product-surface contradiction. It does not block Sprint 25 acceptance because the implementation and tests are sound, but it is a real follow-up item before further CLI expansion.

### Residual risk: docs guardrails are still too narrow for renamed CLI flags

`bun run verify:docs` passed even though the later README CLI Usage block still references the removed manifest spelling. The current guardrails prove the presence of selected canonical strings, but not full consistency across every maintained CLI usage section.

## Product Owner Decision

- Accept Sprint 25 as delivered: the committed backlog items shipped and the merged increment is technically sound.
- Treat CLI config-contract consolidation as the next product hardening priority before any new CLI surface expansion.
- Next intake should explicitly reconcile sprint records and public docs with the shipped `--bulk` / per-target `--config` / `getRenderConfig()` contract, rather than reopening broader CLI design questions.
